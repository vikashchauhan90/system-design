# Comprehensive Guide to GraphQL

## 1. What is GraphQL?

**GraphQL** is a query language for APIs and a runtime for executing those queries against your data. It was developed by Facebook in 2012 and open-sourced in 2015. GraphQL allows clients to request exactly the data they need, nothing more and nothing less, over a single endpoint (usually `/graphql`).

Unlike REST, where multiple endpoints return fixed data structures, GraphQL exposes a **schema** that describes all possible data and operations. Clients then send queries that specify the shape and fields of the response.

---

## 2. Core Concepts

### 2.1 Schema
The schema is the contract between client and server. It defines the types of data and the operations (queries, mutations, subscriptions) available.

```graphql
type Query {
  user(id: ID!): User
  users(limit: Int): [User!]!
}

type Mutation {
  createUser(name: String!, email: String!): User!
}

type Subscription {
  userCreated: User!
}
```

### 2.2 Types
- **Object types** – represent domain objects (e.g., `User`, `Product`).
- **Scalar types** – built-in: `Int`, `Float`, `String`, `Boolean`, `ID`.
- **Enum types** – restricted set of values.
- **Input types** – for mutation arguments.
- **Interface / Union** – for polymorphic results.

### 2.3 Queries
Read operations. Client specifies exactly which fields to return.

```graphql
query {
  user(id: "123") {
    name
    email
    posts {
      title
    }
  }
}
```

### 2.4 Mutations
Write operations (create, update, delete). They have a defined input and return a payload.

```graphql
mutation {
  createUser(name: "Alice", email: "alice@example.com") {
    id
    name
  }
}
```

### 2.5 Subscriptions
Real‑time events over WebSockets. Clients subscribe to events and receive pushed data.

```graphql
subscription {
  userCreated {
    id
    name
  }
}
```

### 2.6 Resolvers
Functions on the server that resolve a field to actual data. Each field in the schema can have a resolver.

## 3. How GraphQL Works

```
Client                        Server
   |                            |
   | 1. Send query (POST /graphql)
   |    { user(id:"123") { name } }
   |--------------------------->|
   |                            | 2. Parse & validate query against schema
   |                            | 3. Execute resolvers (fetch data)
   |                            | 4. Collect result in same shape
   | 5. Response (JSON)         |
   |<---------------------------|
   |    { "data": { "user": { "name": "John" } } }
```

Key steps:
- **Parsing** – text query → abstract syntax tree (AST).
- **Validation** – ensure query matches schema (no unknown fields, correct types).
- **Execution** – call resolvers, starting from root fields.
- **DataLoader** (optimisation) – batching and caching to avoid N+1 queries.

## 4. Pros and Cons

| Pros | Cons |
|------|------|
| **Client‑driven data fetching** – request only needed fields. | **Complexity** – steeper learning curve than REST. |
| **Strongly typed schema** – self‑documenting (introspection). | **Over‑fetching / under‑fetching eliminated** (actually a pro, but requires client discipline). |
| **Single endpoint** – no URL versioning headaches. | **Caching is harder** – traditional HTTP caching less effective (needs persisted queries or CDN). |
| **Real‑time with subscriptions** (WebSockets). | **File uploads** – not standard; require custom scalars or separate endpoint. |
| **Rapid frontend iteration** – frontend teams don't wait for backend changes. | **Performance concerns** – expensive queries (depth limiting, complexity analysis). |
| **Tooling** – GraphiQL, Voyager, Apollo Studio. | **N+1 problem** – requires DataLoader pattern to solve. |
| **Batching** – multiple queries in one request. | **Versioning** – schema evolves by deprecating fields, not versions. |

---

## 5. Use Cases

| Use Case | Why GraphQL? |
|----------|---------------|
| **Mobile apps** (limited bandwidth) | Fetch exactly what the screen needs. |
| **Complex dashboards** (many data sources) | Single endpoint aggregating multiple backends. |
| **Microservices frontend (BFF)** | GraphQL gateway calls downstream services. |
| **Real‑time applications** (chat, live scores) | Subscriptions over WebSockets. |
| **Rapid prototyping** | Frontend drives data requirements. |
| **Third‑party APIs** (public GraphQL APIs) | GitHub, Shopify, etc. |

**When to avoid GraphQL:**
- Simple CRUD APIs with few relationships.
- Applications requiring heavy HTTP caching (e.g., public CDN‑cached content).
- Teams without experience in schema design or resolver optimisation.

## 6. GraphQL vs REST vs RPC

| Feature | GraphQL | REST | RPC (gRPC) |
|---------|---------|------|-------------|
| **Endpoint** | Single `/graphql` | Multiple URLs | Single or multiple |
| **Data fetching** | Client specifies shape | Fixed server‑defined shapes | Function arguments & return |
| **Over‑fetching** | Eliminated | Common | Depends (can be tailored) |
| **Versioning** | Schema evolution (deprecation) | URL or header versioning | Breaking changes require new .proto |
| **Caching** | Harder (needs persisted queries) | HTTP cache easy | Harder (binary) |
| **Real‑time** | Subscriptions (WebSockets) | Webhooks / SSE | gRPC streaming |
| **Performance** | Moderate (text) | Moderate | Very high (binary, HTTP/2) |
| **Browser support** | Native (via fetch) | Native | gRPC‑web required |
| **Code generation** | Optional (TypeScript, Swift, etc.) | OpenAPI tools | Strong (Protobuf) |
| **Learning curve** | Moderate | Low | Moderate |

**Summary:**
- **REST** – public APIs, simple CRUD.
- **GraphQL** – client‑driven apps, complex aggregations.
- **gRPC** – high‑performance microservices, internal systems.

## 7. C# Implementation Examples

### 7.1 Hot Chocolate (GraphQL Server)

Hot Chocolate is the most popular GraphQL server for .NET.

**Install packages:**
```bash
dotnet add package HotChocolate.AspNetCore
dotnet add package HotChocolate.Data
```

**Define model and schema**
```csharp
public class User
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string Email { get; set; }
    public List<Post> Posts { get; set; }
}

public class Post
{
    public int Id { get; set; }
    public string Title { get; set; }
    public int UserId { get; set; }
}
```

**Create Query resolver**
```csharp
public class Query
{
    private readonly List<User> _users = new()
    {
        new User { Id = 1, Name = "Alice", Email = "alice@example.com" }
    };

    public IQueryable<User> GetUsers() => _users.AsQueryable();

    public User? GetUser(int id) => _users.FirstOrDefault(u => u.Id == id);
}
```

**Register GraphQL in Program.cs**
```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Services
    .AddGraphQLServer()
    .AddQueryType<Query>()
    .AddProjections()   // for field selection
    .AddFiltering()     // for where clause
    .AddSorting();      // for orderBy

var app = builder.Build();
app.MapGraphQL();  // endpoint at /graphql
app.Run();
```

**Run and test** – Open `https://localhost:5000/graphql` to access GraphiQL IDE.

**Example query:**
```graphql
query {
  user(id: 1) {
    name
    email
  }
}
```

### 7.2 Mutations (Create User)

```csharp
public class Mutation
{
    private readonly List<User> _users;
    public Mutation(List<User> users) => _users = users;

    public User CreateUser(string name, string email)
    {
        var user = new User { Id = _users.Count + 1, Name = name, Email = email };
        _users.Add(user);
        return user;
    }
}

// Register mutation
builder.Services.AddGraphQLServer()
    .AddQueryType<Query>()
    .AddMutationType<Mutation>();
```

### 7.3 Subscriptions (Real‑time)

```csharp
public class Subscription
{
    [Subscribe]
    public User UserCreated([EventMessage] User user) => user;
}

// In mutation, publish event
public User CreateUser(string name, string email)
{
    var user = new User { ... };
    _users.Add(user);
    // Publish to subscription
    _eventSender.SendAsync("UserCreated", user);
    return user;
}
```

### 7.4 DataLoader to Solve N+1 Problem

Hot Chocolate includes built‑in DataLoader.

```csharp
public class UserResolver
{
    public async Task<User?> GetUserAsync(int id, IResolverContext ctx)
    {
        return await ctx.DataLoader<UserByIdDataLoader>().LoadAsync(id);
    }
}

public class UserByIdDataLoader : BatchDataLoader<int, User>
{
    private readonly IUserRepository _repo;
    public UserByIdDataLoader(IUserRepository repo, IBatchScheduler scheduler, DataLoaderOptions opts)
        : base(scheduler, opts) => _repo = repo;

    protected override async Task<IReadOnlyDictionary<int, User>> LoadBatchAsync(IReadOnlyList<int> keys, CancellationToken ct)
    {
        var users = await _repo.GetUsersByIdsAsync(keys);
        return users.ToDictionary(u => u.Id);
    }
}
```

### 7.5 Client‑side with Strawberry Shake (C#)

Strawberry Shake is a GraphQL client for .NET that generates strongly typed clients.

**Install tool:**
```bash
dotnet new tool-manifest
dotnet tool install StrawberryShake.Tools
```

**Add `.graphql` file to project:**
```graphql
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
    email
  }
}
```

**Build to generate client:**
```bash
dotnet graphql init https://localhost:5000/graphql
dotnet graphql compile
```

**Use generated client:**
```csharp
var client = new MyGraphQLClient(new HttpClient());
var result = await client.GetUser.ExecuteAsync(1);
Console.WriteLine(result.Data.User.Name);
```

## 8. Best Practices

| Area | Recommendation |
|------|----------------|
| **Schema design** | Use meaningful types, avoid exposing raw database models. |
| **Field naming** | camelCase by convention (default in Hot Chocolate). |
| **Error handling** | Use `IErrorFilter` to customise error messages. |
| **Security** | Implement authentication (JWT) and authorisation (policies). |
| **Complexity analysis** | Limit query depth (e.g., 5 levels) and complexity points. |
| **Pagination** | Use `connections` pattern (first, after, edges, node) for lists. |
| **N+1 problem** | Always use DataLoader for batch loading. |
| **Persisted queries** | Pre‑register queries to improve performance and security. |
| **Introspection** | Disable in production (or restrict). |
| **Monitoring** | Use Apollo Studio or Hot Chocolate's built‑in diagnostics. |

**Example of depth limiting:**
```csharp
builder.Services.AddGraphQLServer()
    .AddQueryType<Query>()
    .AddMaxExecutionDepthRule(5);
```

**Example of persisted queries:**
```csharp
builder.Services.AddGraphQLServer()
    .AddQueryType<Query>()
    .AddPersistedQueryProcessor<FileSystemQueryStorage>();
```

---

## 9. Conclusion

GraphQL provides a flexible, efficient alternative to REST for modern applications where client data requirements vary. Its strong typing, introspection, and real‑time subscriptions make it a powerful choice for complex frontends, mobile apps, and API gateways.

**Choose GraphQL when:**
- Clients need different subsets of data.
- You want to reduce over‑fetching and under‑fetching.
- You need real‑time updates.
- Your frontend team iterates rapidly.

**Use Hot Chocolate** for building GraphQL servers in C# – it's feature‑rich, performant, and well‑documented. For clients, **Strawberry Shake** provides type‑safe query execution.
