# Comprehensive Guide to Server-Sent Events (SSE)

## 1. What are Server-Sent Events?

**Server-Sent Events (SSE)** is a standard that enables a server to push data to a client over a single, long‑lived HTTP connection. Unlike WebSockets, SSE is **one‑way**: server → client. The client uses the `EventSource` API (built into modern browsers) to listen for messages.

**Key characteristics:**
- One‑way streaming (server pushes, client receives).
- Text‑based (UTF‑8) – typically JSON.
- Automatic reconnection built into `EventSource`.
- Simple HTTP protocol (no upgrade handshake).
- Supports custom event types and message IDs.

---

## 2. How SSE Works

The client initiates a regular HTTP GET request with the `Accept: text/event-stream` header. The server responds with `Content-Type: text/event-stream` and keeps the connection open indefinitely. The server then sends messages as a stream of UTF‑8 text, each message separated by two newlines (`\n\n`).

**Basic message format:**
```
data: Hello world\n\n
```

**With optional fields:**
```
id: 42
event: userlogin
data: {"username": "alice"}
data: second line of data\n\n
```

**Fields:**
- `data:` – the payload (can be multi‑line).
- `event:` – custom event name (client listens with `addEventListener`).
- `id:` – message identifier (used for reconnection).
- `retry:` – milliseconds to wait before reconnecting.

---

## 3. SSE vs WebSockets vs Long Polling

| Feature | SSE | WebSockets | Long Polling |
|---------|-----|------------|--------------|
| **Direction** | Server → Client only | Full‑duplex | Half‑duplex |
| **Protocol** | HTTP (persistent) | ws:// (upgraded) | HTTP |
| **Binary data** | Not native (Base64) | Native | Base64 |
| **Automatic reconnection** | Built‑in (EventSource) | Manual | Manual |
| **Max connections per domain** (HTTP/1.1) | 6 (browser limit) | Unlimited (theoretically) | 6 |
| **Firewall friendly** | Yes | Sometimes | Yes |
| **Complexity** | Low | Moderate | Low |
| **Browser support** | All modern (except IE) | All modern | All |

**When to choose SSE:**
- One‑way push from server (news feeds, live scores, notifications).
- Simplicity is important.
- You don’t need to send data from client over the same connection.

**When to avoid SSE:**
- Need bi‑directional communication (chat, gaming).
- Binary data (use WebSockets).
- Legacy browser support for IE (polyfills exist).

---

## 4. Use Cases

| Use Case | Why SSE? |
|----------|----------|
| **Live news / stock ticker** | Server pushes updates; client just displays. |
| **Dashboard metrics** | Real‑time charts without complex WebSocket. |
| **Notifications** | One‑way alerts (email, system updates). |
| **Social media feed** | New posts appear automatically. |
| **Progress updates** (file upload, long job) | Server pushes progress percentage. |
| **Server log streaming** | Live tailing of logs. |

## 5. C# Implementation Examples

### 5.1 Basic SSE Endpoint in ASP.NET Core

**Server (Program.cs)**
```csharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/sse", async (HttpContext context) =>
{
    context.Response.Headers.Add("Content-Type", "text/event-stream");
    context.Response.Headers.Add("Cache-Control", "no-cache");
    context.Response.Headers.Add("Connection", "keep-alive");

    var i = 0;
    while (!context.RequestAborted.IsCancellationRequested)
    {
        await context.Response.WriteAsync($"data: Hello {i}\n\n");
        await context.Response.Body.FlushAsync();
        await Task.Delay(1000);
        i++;
    }
});

app.Run();
```

**JavaScript client**
```javascript
const source = new EventSource("/sse");
source.onmessage = (event) => {
    console.log("Received:", event.data);
};
```

### 5.2 SSE with Periodic Data Push (JSON)

**Server sending JSON**
```csharp
app.MapGet("/stock-prices", async (HttpContext context) =>
{
    context.Response.Headers.Add("Content-Type", "text/event-stream");
    context.Response.Headers.Add("Cache-Control", "no-cache");

    var random = new Random();
    while (!context.RequestAborted.IsCancellationRequested)
    {
        var price = new { symbol = "MSFT", price = random.Next(100, 200) };
        var json = JsonSerializer.Serialize(price);
        await context.Response.WriteAsync($"data: {json}\n\n");
        await context.Response.Body.FlushAsync();
        await Task.Delay(2000);
    }
});
```

**Client**
```javascript
const source = new EventSource("/stock-prices");
source.onmessage = (e) => {
    const data = JSON.parse(e.data);
    console.log(`${data.symbol}: $${data.price}`);
};
```

### 5.3 SSE with Real‑Time Notifications (Using Background Service)

**Notification service (singleton)**
```csharp
public class NotificationService
{
    private readonly List<HttpContext> _clients = new();
    private readonly object _lock = new();

    public void AddClient(HttpContext context)
    {
        lock (_lock) _clients.Add(context);
    }

    public void RemoveClient(HttpContext context)
    {
        lock (_lock) _clients.Remove(context);
    }

    public async Task BroadcastAsync(string message)
    {
        List<HttpContext> clientsCopy;
        lock (_lock) clientsCopy = _clients.ToList();

        foreach (var client in clientsCopy)
        {
            try
            {
                await client.Response.WriteAsync($"data: {message}\n\n");
                await client.Response.Body.FlushAsync();
            }
            catch { RemoveClient(client); }
        }
    }
}
```

**Endpoint**
```csharp
app.MapGet("/notifications", async (HttpContext context, NotificationService ns) =>
{
    context.Response.Headers.Add("Content-Type", "text/event-stream");
    context.Response.Headers.Add("Cache-Control", "no-cache");

    ns.AddClient(context);
    context.RequestAborted.Register(() => ns.RemoveClient(context));

    // Keep connection open
    await Task.Delay(-1, context.RequestAborted);
});
```

**Trigger notification from another endpoint**
```csharp
app.MapPost("/notify", async (string msg, NotificationService ns) =>
{
    await ns.BroadcastAsync(msg);
    return Results.Ok();
});
```

### 5.4 JavaScript Client with Custom Events

```javascript
const source = new EventSource("/events");
source.addEventListener("user-login", (e) => {
    console.log("User login:", e.data);

```

**Server sending custom event**
```csharp
await context.Response.WriteAsync($"event: user-login\ndata: {{\"user\":\"alice\"}}\n\n");
```

### 5.5 C# Client for SSE (using `System.Net.Http`)

```csharp
using var client = new HttpClient();
var request = new HttpRequestMessage(HttpMethod.Get, "http://localhost:5000/sse");
request.Headers.Add("Accept", "text/event-stream");

using var response = await client.SendAsync(request, HttpCompletionOption.ResponseHeadersRead);
await using var stream = await response.Content.ReadAsStreamAsync();
using var reader = new StreamReader(stream);

while (!reader.EndOfStream)
{
    var line = await reader.ReadLineAsync();
    if (line?.StartsWith("data:") == true)
    {
        var data = line.Substring(5).Trim();
        Console.WriteLine(data);
    }
}
```

## 6. Advanced Features

### 6.1 Custom Event Types

Send different event names so client can listen separately.

**Server**
```csharp
await context.Response.WriteAsync("event: message\ndata: Hello\n\n");
await context.Response.WriteAsync("event: ping\ndata: keepalive\n\n");
```

**Client**
```javascript
source.addEventListener("message", (e) => console.log(e.data));
source.addEventListener("ping", (e) => console.log("Ping received"));
```

### 6.2 Retry Mechanism

The client will automatically reconnect after a delay. The server can suggest a retry time (in milliseconds).

```csharp
await context.Response.WriteAsync("retry: 10000\n\n"); // 10 seconds
```

### 6.3 Message ID for Reliable Delivery

When reconnecting, the client sends the last received `id` in the `Last-Event-ID` header. The server can then resume from that ID.

**Server**
```csharp
int messageId = 0;
while (true)
{
    await context.Response.WriteAsync($"id: {messageId}\ndata: Update {messageId}\n\n");
    messageId++;
    await Task.Delay(1000);
}
```

## 7. Security Considerations

| Threat | Mitigation |
|--------|-------------|
| **Cross‑origin requests** | Use CORS policies. `EventSource` respects CORS. |
| **Authentication** | Pass token via query string (e.g., `EventSource("/sse?token=...")`). Validate on server. |
| **Denial of Service** | Limit number of concurrent SSE connections per user. Set timeouts. |
| **Sensitive data** | Use `wss://` (HTTPS) to encrypt the stream. |
| **Message injection** | Validate and sanitise data before sending. |

**Authentication example**
```csharp
app.MapGet("/sse", async (HttpContext context) =>
{
    var token = context.Request.Query["token"];
    if (!IsValidToken(token)) 
    {
        context.Response.StatusCode = 401;
        return;
    }
    // proceed with SSE
});
```

## 8. Scaling SSE

SSE connections are long‑lived and stateful. To scale across multiple servers:

- **Use sticky sessions (affinity)** – load balancer routes a client to the same server.
- **Use a backplane** – broadcast messages to all servers (e.g., Redis pub/sub). Each server forwards messages to its connected clients.
- **Azure SignalR Service** (also supports SSE fallback).

**Simpler approach:** Use Redis to publish messages; each SSE endpoint subscribes to Redis and forwards to its local clients.

## 9. Best Practices

| Area | Recommendation |
|------|----------------|
| **Heartbeat** | Send a periodic comment (`:\n\n`) to prevent idle timeouts (proxies, browsers). |
| **Compression** | Use HTTPS (which supports compression) or gzip the stream. |
| **Message size** | Keep small (< 64KB). For large data, send a reference. |
| **Connection limit** | Browser limits to 6 per domain (HTTP/1.1). Use HTTP/2 or subdomains for more. |
| **Error handling** | On client, listen to `onerror` and reconnect manually if needed. |
| **Logging** | Log connection opens/closes and errors, not message content. |
| **Test** | Use `curl` for quick testing: `curl -H "Accept:text/event-stream" http://localhost/sse` |

**Heartbeat (comment)**
```csharp
await context.Response.WriteAsync(":\n\n"); // no data, just keeps connection alive
```

**Client error handling**
```javascript
source.onerror = (e) => {
    console.error("SSE connection lost. Reconnecting...");
    // EventSource automatically reconnects, but you can log or retry with backoff.
};
```

---

## 10. Conclusion

Server‑Sent Events are a simple yet powerful technology for one‑way, real‑time server push. They are ideal for live updates, notifications, dashboards, and any scenario where the client only needs to receive data. SSE offers automatic reconnection, low overhead, and easy implementation in ASP.NET Core.

**Choose SSE when:**
- You need real‑time push from server to client.
- Bi‑directional communication is not required.
- You want simplicity and browser compatibility.
- Your application can tolerate text‑only payloads.

**Avoid SSE when:**
- You need full‑duplex (use WebSockets).
- Binary data is frequent (use WebSockets).
- You require legacy IE support (polyfills exist, but WebSockets may be better).
