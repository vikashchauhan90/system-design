# Comprehensive Guide to WebSockets

## 1. What are WebSockets?

WebSocket is a full‑duplex communication protocol over a single, persistent TCP connection. It enables real‑time, low‑latency, bi‑directional data exchange between a client and server. Unlike HTTP, which is request‑response, WebSocket allows both endpoints to send messages at any time after the initial handshake.

**Key characteristics:**
- Full‑duplex (both sides can send independently).
- Persistent connection (no repeated handshakes).
- Low overhead (no HTTP headers per message).
- Binary or text messages.
- Works over port 80 (ws://) or 443 (wss://).

---

## 2. How WebSockets Work

The connection starts with an HTTP GET request that includes an `Upgrade: websocket` header. The server responds with `101 Switching Protocols`. Once upgraded, the same TCP socket is used for all subsequent messages.

**Handshake example (client request):**
```http
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Protocol: chat
Sec-WebSocket-Version: 13
```

**Server response:**
```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

After the handshake, messages are sent as **frames** (text or binary) with minimal overhead (2–14 bytes).

---

## 3. WebSocket vs HTTP vs Other Real‑Time Technologies

| Feature | WebSocket | HTTP | Server‑Sent Events (SSE) | Long Polling |
|---------|-----------|------|--------------------------|--------------|
| **Direction** | Full‑duplex | Half‑duplex | Server → Client only | Half‑duplex |
| **Latency** | Very low | High (per request) | Low | Low |
| **Overhead** | Low (frame headers) | High (HTTP headers) | Low | Moderate |
| **Binary data** | Native | Base64 (or raw) | Not native | Base64 |
| **Browser support** | All modern | All | Almost all | All |
| **Firewall friendly** | Sometimes (upgrade may be blocked) | Yes | Yes | Yes |
| **Automatic reconnection** | Manual (library helps) | N/A | Built‑in (EventSource) | Manual |

**When to choose WebSockets:**
- Need bi‑directional real‑time communication (chat, gaming, live collaboration).
- Low latency and high throughput are critical.
- Binary data (audio, video, protobuf) is involved.

## 5. C# Implementation Examples

### 5.1 Raw WebSocket in ASP.NET Core

**Server (Program.cs)**
```csharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.UseWebSockets();

app.Map("/ws", async context =>
{
    if (!context.WebSockets.IsWebSocketRequest)
    {
        context.Response.StatusCode = 400;
        return;
    }

    using var webSocket = await context.WebSockets.AcceptWebSocketAsync();
    await Echo(webSocket);
});

async Task Echo(WebSocket webSocket)
{
    var buffer = new byte[1024];
    while (webSocket.State == WebSocketState.Open)
    {
        var result = await webSocket.ReceiveAsync(new ArraySegment<byte>(buffer), CancellationToken.None);
        if (result.MessageType == WebSocketMessageType.Close)
        {
            await webSocket.CloseAsync(WebSocketCloseStatus.NormalClosure, "", CancellationToken.None);
        }
        else
        {
            // Echo back
            await webSocket.SendAsync(new ArraySegment<byte>(buffer, 0, result.Count), result.MessageType, true, CancellationToken.None);
        }
    }
}

app.Run();
```

**JavaScript client**
```javascript
const ws = new WebSocket("ws://localhost:5000/ws");
ws.onopen = () => ws.send("Hello Server");
ws.onmessage = (event) => console.log("Received:", event.data);
```

### 5.2 SignalR (Recommended for Production)

SignalR abstracts WebSockets (with fallbacks to long polling, SSE). Install `Microsoft.AspNetCore.SignalR`.

**Server Hub**
```csharp
public class ChatHub : Hub
{
    public async Task SendMessage(string user, string message)
    {
        await Clients.All.SendAsync("ReceiveMessage", user, message);
    }
}

// Program.cs
builder.Services.AddSignalR();
app.MapHub<ChatHub>("/chatHub");
```

**JavaScript client**
```javascript
const connection = new signalR.HubConnectionBuilder()
    .withUrl("/chatHub")
    .build();

connection.on("ReceiveMessage", (user, msg) => console.log(user, msg));
await connection.start();
await connection.invoke("SendMessage", "Alice", "Hello");
```

**C# client**
```csharp
using Microsoft.AspNetCore.SignalR.Client;

await using var connection = new HubConnectionBuilder()
    .WithUrl("https://localhost:5000/chatHub")
    .Build();

connection.On<string, string>("ReceiveMessage", (user, msg) => Console.WriteLine($"{user}: {msg}"));
await connection.StartAsync();
await connection.InvokeAsync("SendMessage", "Bob", "Hi");
```

**SignalR advantages:**
- Automatic reconnection.
- Fallback to long polling / SSE.
- Scaling with Redis backplane.
- Simple API (send/receive).

### 5.3 WebSocket Client in C# (without SignalR)

```csharp
using System.Net.WebSockets;
using System.Text;

using var ws = new ClientWebSocket();
await ws.ConnectAsync(new Uri("ws://localhost:5000/ws"), CancellationToken.None);

var sendBuffer = Encoding.UTF8.GetBytes("Hello");
await ws.SendAsync(new ArraySegment<byte>(sendBuffer), WebSocketMessageType.Text, true, CancellationToken.None);

var receiveBuffer = new byte[1024];
var result = await ws.ReceiveAsync(new ArraySegment<byte>(receiveBuffer), CancellationToken.None);
var message = Encoding.UTF8.GetString(receiveBuffer, 0, result.Count);
Console.WriteLine(message);
```

## 6. Security Considerations

| Threat | Mitigation |
|--------|-------------|
| **Man‑in‑the‑middle** | Always use `wss://` (TLS). Same certificate as HTTPS. |
| **Cross‑site WebSocket hijacking** | Authenticate during handshake (e.g., JWT in query string or `Sec-WebSocket-Protocol`). |
| **Denial of service** | Limit connection rate, message size, and frame rate. Implement timeouts. |
| **Input validation** | Validate all messages; never trust client input. |
| **Origin check** | On the server, verify `Origin` header against allowed list. |

**Authentication example (with SignalR):**
```csharp
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Events = new JwtBearerEvents
        {
            OnMessageReceived = context =>
            {
                var accessToken = context.Request.Query["access_token"];
                var path = context.HttpContext.Request.Path;
                if (!string.IsNullOrEmpty(accessToken) && path.StartsWithSegments("/chatHub"))
                    context.Token = accessToken;
                return Task.CompletedTask;
            }
        };
    });
```

## 7. Scaling WebSockets

WebSocket connections are stateful (persistent). To scale across multiple servers, you need a **backplane** (e.g., Redis) to synchronise messages.

**SignalR Redis backplane:**
```bash
dotnet add package Microsoft.AspNetCore.SignalR.StackExchangeRedis
```

```csharp
builder.Services.AddSignalR().AddStackExchangeRedis("redis:6379");
```

**Alternative:** Use Azure SignalR Service or stick to a single server with sticky sessions (load balancer affinity).

## 8. Best Practices

| Area | Recommendation |
|------|----------------|
| **Message format** | Use a consistent serialisation (JSON, MessagePack, Protobuf). |
| **Heartbeat / ping** | Send periodic `ping` frames or small messages to detect dead connections. |
| **Reconnection** | Implement exponential backoff. SignalR does this automatically. |
| **Message size** | Keep < 64KB; for larger data, send reference and fetch via HTTP. |
| **Compression** | Enable `permessage-deflate` for text messages (reduces bandwidth). |
| **Logging** | Log connection open/close and errors, but not message content. |
| **Testing** | Use tools like `wscat`, Postman WebSocket requests. |
| **Fallbacks** | Provide a fallback (long polling) for environments that block WebSockets. SignalR does this. |

**Heartbeat example (raw WebSocket server):**
```csharp
var cts = new CancellationTokenSource();
var heartbeat = Task.Run(async () =>
{
    while (!cts.IsCancellationRequested)
    {
        await Task.Delay(30000);
        if (webSocket.State == WebSocketState.Open)
            await webSocket.SendAsync(new ArraySegment<byte>(Encoding.UTF8.GetBytes("ping")), WebSocketMessageType.Text, true, CancellationToken.None);
    }
});
```

---

## 9. Conclusion

WebSockets enable true real‑time, full‑duplex communication. For .NET developers, **SignalR** is the recommended approach – it provides a high‑level API, automatic fallbacks, and scaling support. Raw `System.Net.WebSockets` is available for low‑level control.

**Choose WebSockets when:**
- You need bi‑directional, low‑latency communication.
- You can accept the complexity of persistent connections.
- You have a compatible infrastructure (load balancers that support WebSockets).

**Avoid WebSockets for:**
- Simple request‑response (use REST).
- Infrequent updates (use polling or SSE).
