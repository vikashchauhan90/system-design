# Asynchronous Operations (202 Accepted + Webhooks)

### Introduction
Long‑running operations (e.g., generating reports, processing videos) cannot complete within typical HTTP timeouts. The **202 Accepted** pattern lets the server acknowledge the request and return a status URL for polling. Alternatively, **webhooks** notify the client when the operation finishes.

### 202 Accepted Pattern

**Best Practices**
- Return `202 Accepted` immediately with a `Location` header pointing to a status endpoint.
- Include an estimated completion time in the response body or headers.
- The status endpoint returns `200 OK` when complete (with result) or `202` if still processing.
- Allow cancellation using `DELETE` on the status URL.

**C# Implementation**

**Request endpoint**
```csharp
[HttpPost("reports")]
public IActionResult GenerateReport([FromBody] ReportRequest request)
{
    var jobId = Guid.NewGuid().ToString();
    // Queue the job (e.g., Hangfire, BackgroundService)
    _backgroundQueue.QueueJob(jobId, request);
    var statusUrl = Url.Action(nameof(GetJobStatus), new { jobId });
    return Accepted(statusUrl, new { jobId, statusUrl, estimatedCompletion = DateTime.UtcNow.AddMinutes(5) });
}
```

**Status endpoint**
```csharp
[HttpGet("jobs/{jobId}")]
public IActionResult GetJobStatus(string jobId)
{
    var job = _jobStore.Get(jobId);
    if (job == null) return NotFound();

    if (job.IsCompleted)
    {
        return Ok(job.Result); // 200 with the final result
    }
    else if (job.IsFailed)
    {
        return StatusCode(500, new { error = job.Error });
    }
    else
    {
        return Accepted(new { status = "processing", progress = job.Progress });
    }
}
```

### Webhooks

**Description**  
Client provides a callback URL. The server sends a POST request to that URL when the operation finishes. This eliminates polling.

**Best Practices**
- Clients register a webhook URL (e.g., `POST /api/webhooks`) with a secret for signing.
- Secure webhooks: verify the signature using a shared secret.
- Retry on failure (exponential backoff).
- Idempotency: include an `X-Event-Id` header to deduplicate.

**C# Implementation (Client side: registering webhook)**
```csharp
[HttpPost("webhooks/register")]
public IActionResult RegisterWebhook([FromBody] WebhookRegistration reg)
{
    // Store reg.Url and reg.Secret for the user
    return Ok();
}
```

**Server side: sending webhook notification**
```csharp
public async Task SendWebhookAsync(string url, string secret, object payload)
{
    var json = JsonSerializer.Serialize(payload);
    var signature = ComputeHmacSha256(json, secret);
    using var client = new HttpClient();
    var request = new HttpRequestMessage(HttpMethod.Post, url)
    {
        Content = new StringContent(json, Encoding.UTF8, "application/json")
    };
    request.Headers.Add("X-Webhook-Signature", signature);
    request.Headers.Add("X-Event-Id", Guid.NewGuid().ToString());
    var response = await client.SendAsync(request);
    if (!response.IsSuccessStatusCode)
    {
        // Retry later (e.g., add to a retry queue)
    }
}

private string ComputeHmacSha256(string payload, string secret)
{
    var hmac = new HMACSHA256(Encoding.UTF8.GetBytes(secret));
    var hash = hmac.ComputeHash(Encoding.UTF8.GetBytes(payload));
    return Convert.ToBase64String(hash);
}
```

**Webhook Payload Example**
```json
{
  "event": "report.generated",
  "timestamp": "2025-04-02T10:30:00Z",
  "data": {
    "jobId": "abc123",
    "downloadUrl": "/reports/abc123.pdf"
  }
}
```

**Polling vs Webhooks**
| Polling (202 + status) | Webhooks |
|------------------------|----------|
| Simpler for client | More efficient (no polling) |
| Guaranteed eventual result | Requires public callback URL |
| Higher server load if many clients | Harder to debug |

**Combined approach:** Offer both – clients can poll or provide a webhook.
