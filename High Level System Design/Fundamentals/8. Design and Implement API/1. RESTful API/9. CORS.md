# Comprehensive Guide to CORS for Web APIs

## 1. What is CORS?

**Cross-Origin Resource Sharing (CORS)** is a security mechanism implemented by web browsers that allows a web page from one origin (domain, protocol, port) to request resources from a different origin. Without CORS, browsers enforce the **Same-Origin Policy** (SOP), which blocks such requests.

**Origin** = scheme (http/https) + hostname (domain) + port (if specified).  
Example: `https://myapp.com:443` is a different origin than `http://myapp.com` (different scheme) or `https://api.myapp.com` (different hostname).

CORS is **not** a security feature to protect your API from malicious requests – it’s a way for the server to **relax** the same‑origin policy for trusted clients.

---

## 2. Why is CORS Needed?

Modern web applications often have frontend and backend on different origins:
- Frontend: `https://myfrontend.com`
- Backend API: `https://api.mybackend.com`

Without CORS, the browser would block the frontend’s JavaScript `fetch` or `XMLHttpRequest` to the backend. CORS headers tell the browser that the backend allows the frontend’s origin.

**Important:** CORS only applies to browser‑based requests. Mobile apps, desktop apps, and server‑to‑server calls are **not** affected by CORS because they don’t enforce the same‑origin policy.

---

## 3. How CORS Works (Preflight Requests)

For certain requests (those with custom headers, non‑simple methods like PUT/DELETE, or content‑types other than `application/x-www-form-urlencoded`, `multipart/form-data`, `text/plain`), the browser first sends a **preflight** request using the `OPTIONS` method. The preflight asks the server: “Is the actual request allowed?”

**Preflight request example:**
```http
OPTIONS /api/data HTTP/1.1
Origin: https://myfrontend.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: content-type, authorization
```

**Server must respond with appropriate CORS headers:**
```http
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://myfrontend.com
Access-Control-Allow-Methods: POST, GET, OPTIONS
Access-Control-Allow-Headers: content-type, authorization
Access-Control-Max-Age: 86400
```

If the preflight succeeds, the browser then sends the actual request.

**Simple requests** (GET with certain headers, no custom content‑type) do **not** trigger a preflight. However, the response must still include `Access-Control-Allow-Origin`.

## 4. CORS Configuration in ASP.NET Core

In ASP.NET Core, CORS is configured via the `AddCors` and `UseCors` middleware.

### 4.1 Basic Setup (Allow Any – NOT for production)

```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowAll",
        policy =>
        {
            policy.AllowAnyOrigin()
                  .AllowAnyMethod()
                  .AllowAnyHeader();
        });
});

var app = builder.Build();
app.UseCors("AllowAll");
app.MapControllers();
app.Run();
```

**Warning:** `AllowAnyOrigin` cannot be combined with `AllowCredentials()`. Also, it’s insecure for production because any website can call your API.

### 4.2 Allow Specific Origins

```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowSpecificOrigin",
        policy =>
        {
            policy.WithOrigins("https://myfrontend.com", "https://staging.myfrontend.com")
                  .AllowAnyMethod()
                  .AllowAnyHeader();
        });
});
```

### 4.3 Allow Specific HTTP Methods

```csharp
policy.WithOrigins("https://myfrontend.com")
      .WithMethods("GET", "POST", "PUT")
      .AllowAnyHeader();
```

### 4.4 Allow Credentials (Cookies, Authorization Headers)

When the frontend sends credentials (cookies, HTTP authentication, or client certificates), the server must explicitly allow credentials and **cannot** use `AllowAnyOrigin` (must specify exact origins).

**Server:**
```csharp
policy.WithOrigins("https://myfrontend.com")
      .AllowCredentials()
      .AllowAnyMethod()
      .AllowAnyHeader();
```

**Client (JavaScript with credentials):**
```javascript
fetch('https://api.example.com/data', {
    credentials: 'include'  // sends cookies
});
```

### 4.5 Expose Custom Headers

By default, browsers only expose a few response headers. If your API returns custom headers (e.g., `X-Total-Count`), you must explicitly expose them:

```csharp
policy.WithOrigins("https://myfrontend.com")
      .WithExposedHeaders("X-Total-Count", "X-Custom-Header");
```

### 4.6 Preflight Max Age (Caching)

To reduce preflight requests, set a maximum age (in seconds) for caching the preflight result:

```csharp
policy.WithOrigins("https://myfrontend.com")
      .SetPreflightMaxAge(TimeSpan.FromSeconds(86400)); // 24 hours
```

## 5. Advanced Scenarios

### 5.1 CORS with Authentication (JWT)

If your API uses JWT tokens sent in the `Authorization` header, you don’t need `AllowCredentials()` unless you also send cookies. The `Authorization` header is already allowed by `AllowAnyHeader()` or `WithHeaders("authorization")`.

**Server configuration:**
```csharp
policy.WithOrigins("https://myfrontend.com")
      .AllowAnyMethod()
      .AllowAnyHeader(); // includes Authorization
```

**Client (with token):**
```javascript
fetch('https://api.example.com/data', {
    headers: { 'Authorization': `Bearer ${token}` }
});
```

### 5.2 CORS with SignalR / WebSockets

SignalR uses CORS for the initial HTTP handshake, but WebSocket connections have their own origin checking. Configure CORS to allow the SignalR hub:

```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("SignalRCors", policy =>
    {
        policy.WithOrigins("https://myfrontend.com")
              .AllowAnyMethod()
              .AllowAnyHeader()
              .AllowCredentials(); // SignalR requires credentials
    });
});
app.UseCors("SignalRCors");
app.MapHub<ChatHub>("/chatHub");
```

### 5.3 Policy per Controller or Action

Instead of a global policy, you can apply CORS attributes to specific controllers or actions.

**Define a named policy:**
```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowMobileApp", policy =>
    {
        policy.WithOrigins("https://mobileapp.com")
              .AllowAnyMethod()
              .AllowAnyHeader();
    });
});
```

**Apply to controller:**
```csharp
[EnableCors("AllowMobileApp")]
[ApiController]
[Route("api/[controller]")]
public class DataController : ControllerBase { ... }
```

**Disable CORS for an action:**
```csharp
[DisableCors]
[HttpGet("public")]
public IActionResult Public() => Ok("No CORS needed");
```

## 6. Testing CORS

### 6.1 Using JavaScript `fetch`

Open browser console on your frontend origin (e.g., `https://myfrontend.com`) and run:

```javascript
fetch('https://api.example.com/api/data', {
    method: 'GET',
    headers: { 'Content-Type': 'application/json' }
})
.then(res => res.json())
.then(data => console.log(data))
.catch(err => console.error(err));
```

If CORS is misconfigured, you’ll see an error like:  
`Access to fetch at 'https://api.example.com/api/data' from origin 'https://myfrontend.com' has been blocked by CORS policy`.

### 6.2 Using `curl`

`curl` does not enforce CORS, but you can inspect CORS headers:

```bash
curl -i -X OPTIONS https://api.example.com/api/data \
  -H "Origin: https://myfrontend.com" \
  -H "Access-Control-Request-Method: GET" \
  -H "Access-Control-Request-Headers: content-type"
```

Check the response for `Access-Control-Allow-Origin` and other CORS headers.

## 7. Best Practices

| Practice | Recommendation |
|----------|----------------|
| **Never use `AllowAnyOrigin` in production** | Specify exact origins. For development, use environment‑specific configuration. |
| **Use HTTPS everywhere** | CORS works over HTTPS; avoid mixed content. |
| **Limit allowed methods** | Only allow what your API actually uses (e.g., `GET`, `POST`). |
| **Limit allowed headers** | Only allow headers your API expects (e.g., `content-type`, `authorization`). |
| **Do not use `AllowCredentials` with `AllowAnyOrigin`** – it’s not allowed by the spec and will cause errors. | |
| **Set preflight max age** to reduce latency for repeated requests. | |
| **Log CORS failures** – enable CORS diagnostic logging during development. | |
| **Use policy names** for clarity and reusability. | |

**Example production‑ready configuration:**
```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("ProductionCors", policy =>
    {
        policy.WithOrigins("https://myfrontend.com", "https://admin.myfrontend.com")
              .WithMethods("GET", "POST", "PUT", "DELETE")
              .WithHeaders("content-type", "authorization")
              .WithExposedHeaders("X-Total-Count")
              .SetPreflightMaxAge(TimeSpan.FromHours(1));
    });
});
```

## 8. Common Errors & Troubleshooting

| Error | Likely Cause | Solution |
|-------|--------------|----------|
| `No 'Access-Control-Allow-Origin' header is present` | Server not configured for CORS, or origin not allowed. | Enable CORS and add the requesting origin. |
| `The value of the 'Access-Control-Allow-Origin' header must not be '*'` when using credentials | `AllowAnyOrigin` combined with `AllowCredentials`. | Use specific origins instead of `*`. |
| `Request header field X-Custom is not allowed` | The header is not included in `AllowAnyHeader` or `WithHeaders`. | Add the header to `WithHeaders()`. |
| `Method PUT is not allowed` | The method is not in `AllowAnyMethod` or `WithMethods`. | Add the method to the policy. |
| Preflight request fails (404) | The server does not handle `OPTIONS` requests. | Ensure CORS middleware is registered **before** other middleware that might short‑circuit (e.g., authentication). |
| CORS works for GET but not for POST with JSON | POST with JSON is a “non‑simple” request and requires a preflight. | Ensure your CORS policy includes the `content-type` header and the `OPTIONS` method. |

**Middleware order is critical:**  
`UseCors` must be placed **before** `UseAuthentication` and `UseAuthorization` (if you want CORS to apply to all endpoints, including those that require auth). Also before `UseResponseCaching` or `UseStaticFiles` if they might handle requests.

```csharp
app.UseCors("PolicyName");
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();
```

---

## 9. Conclusion

CORS is essential for modern web applications where frontend and backend are hosted on different origins. ASP.NET Core provides flexible, easy‑to‑configure CORS middleware. Always:

- Specify exact origins in production.
- Limit methods and headers.
- Understand preflight requests.
- Place CORS middleware early in the pipeline.
- Test thoroughly using browser developer tools and `curl`.

By following the patterns and examples in this document, you can secure your Web API while enabling legitimate cross‑origin requests from trusted frontends.
