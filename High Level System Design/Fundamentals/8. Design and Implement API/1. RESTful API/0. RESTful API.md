# Comprehensive Guide to RESTful APIs (including Idempotency with Cache Example)


## 1. Introduction to RESTful APIs <a name="introduction"></a>

**REST** (Representational State Transfer) is an architectural style for designing networked applications. A **RESTful API** follows REST principles, using HTTP as the transport protocol. It allows clients to interact with server resources via standard HTTP methods, returning representations of resource states (e.g., JSON, XML).

**Representational State** means a snapshot of a resource’s data (state) in a format (representation) that the client can understand. For example, a `GET /users/123` request returns a representation of that user resource.

**Benefits of RESTful APIs:**
- **Scalability:** Statelessness and caching reduce server load.
- **Flexibility:** Clients and servers can be implemented in different technologies.
- **Independence:** Decoupled client-server evolution.
- **Uniform interface:** Standard HTTP methods and status codes simplify integration.


## 2. Key Principles of REST <a name="key-principles"></a>

| Principle | Description |
|-----------|-------------|
| **Statelessness** | Each request must contain all necessary information; the server does not store client context. |
| **Client-Server Separation** | UI and data storage concerns are separated. |
| **Cacheability** | Responses explicitly indicate cacheability. |
| **Layered System** | Clients cannot tell if they are connected directly to the end server or an intermediary. |
| **Uniform Interface** | Consistent use of HTTP methods, resource identifiers (URLs), and representations. |
| **Code on Demand (optional)** | Server can send executable code to extend client functionality. |

**How a RESTful API works:**
1. **Client sends a request** – HTTP method, URL, headers, optional body.
2. **Server authenticates and processes** – validates and performs action.
3. **Server sends a response** – status code, headers, optional body (representation).


## 3. HTTP Methods (Verbs) <a name="http-methods"></a>

### GET
- **Purpose:** Retrieve a resource.
- **Safe:** Yes | **Idempotent:** Yes | **Body:** Not allowed
- **Example:** `GET /api/products/123`

### POST
- **Purpose:** Create a new resource.
- **Safe:** No | **Idempotent:** No | **Body:** Representation of new resource
- **Example:** `POST /api/products`

### PUT
- **Purpose:** Replace an entire resource (or create if not exists).
- **Safe:** No | **Idempotent:** Yes | **Body:** Full representation
- **Example:** `PUT /api/products/123`

### PATCH
- **Purpose:** Partially update a resource.
- **Safe:** No | **Idempotent:** Not guaranteed | **Body:** Partial instructions
- **Example:** `PATCH /api/products/123`

### DELETE
- **Purpose:** Remove a resource.
- **Safe:** No | **Idempotent:** Yes | **Body:** Usually empty
- **Example:** `DELETE /api/products/123`

### HEAD, OPTIONS, TRACE
- **HEAD:** Same as GET without body (idempotent, safe).
- **OPTIONS:** Describe allowed methods (idempotent, safe).
- **TRACE:** Echo request for debugging (rarely used).


## 4. HTTP Status Codes <a name="status-codes"></a>

| Category | Example Codes | Meaning |
|----------|--------------|---------|
| **1xx Informational** | 100, 101 | Request received, continuing. |
| **2xx Success** | 200 OK, 201 Created, 204 No Content | Success. |
| **3xx Redirection** | 301 Moved, 304 Not Modified | Further action needed. |
| **4xx Client Error** | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 422 Unprocessable Entity | Client error. |
| **5xx Server Error** | 500 Internal Server Error, 503 Service Unavailable | Server error. |


## 5. Idempotency and Safety <a name="idempotency"></a>

- **Safe methods:** Do not change server state (GET, HEAD, OPTIONS).
- **Idempotent methods:** Same effect even after multiple identical calls (GET, PUT, DELETE, HEAD, OPTIONS, TRACE). POST is **not** idempotent.

**Why idempotency matters:** Network issues may cause duplicate requests. With idempotent methods, retries are safe. For non‑idempotent POST requests, we must implement idempotency manually using an **idempotency key**.


## 6. Idempotency Implementation with Cache (Redis / In‑Memory) <a name="idempotency-implementation"></a>

To make a POST endpoint idempotent, clients send a unique `Idempotency-Key` header (e.g., UUID). The server stores the response for that key. If the same key arrives again with the same request path and method, the server returns the cached response instead of processing the request again.

We use a hash of `HTTP method + request path + idempotency key` as the cache key. This prevents collisions across different endpoints.

### 6.1 In‑Memory Cache Example (using IMemoryCache)

```csharp
public class IdempotencyService
{
    private readonly IMemoryCache _cache;
    public IdempotencyService(IMemoryCache cache) => _cache = cache;

    public string GenerateCacheKey(HttpRequest request, string idempotencyKey)
    {
        // Combine method, path, and idempotency key then hash
        var raw = $"{request.Method}|{request.Path}|{idempotencyKey}";
        using var sha = System.Security.Cryptography.SHA256.Create();
        var hash = sha.ComputeHash(Encoding.UTF8.GetBytes(raw));
        return Convert.ToBase64String(hash);
    }

    public bool TryGetResponse(string cacheKey, out object response)
    {
        return _cache.TryGetValue(cacheKey, out response);
    }

    public void StoreResponse(string cacheKey, object response, TimeSpan ttl)
    {
        _cache.Set(cacheKey, response, ttl);
    }
}
```

**Usage in a controller:**

```csharp
[HttpPost("orders")]
public async Task<IActionResult> CreateOrder([FromBody] OrderRequest request)
{
    // 1. Read idempotency key from header
    if (!Request.Headers.TryGetValue("Idempotency-Key", out var keyHeader))
        return BadRequest("Missing Idempotency-Key header");
    var idempotencyKey = keyHeader.ToString();

    // 2. Generate cache key based on method, path, and key
    var cacheKey = _idempotencyService.GenerateCacheKey(Request, idempotencyKey);

    // 3. Check if already processed
    if (_idempotencyService.TryGetResponse(cacheKey, out var cachedResponse))
        return Ok(cachedResponse); // Return same response as before

    // 4. Process the request (e.g., create order)
    var result = await _orderService.CreateOrderAsync(request);

    // 5. Store the response in cache (e.g., 24 hours)
    _idempotencyService.StoreResponse(cacheKey, result, TimeSpan.FromHours(24));

    return Ok(result);
}
```

### 6.2 Redis Distributed Cache Example

Install `Microsoft.Extensions.Caching.StackExchangeRedis`. Register Redis in `Program.cs`:

```csharp
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = "localhost:6379";
    options.InstanceName = "Idempotency_";
});
```

Then inject `IDistributedCache`:

```csharp
public class RedisIdempotencyService
{
    private readonly IDistributedCache _cache;
    public RedisIdempotencyService(IDistributedCache cache) => _cache = cache;

    public string GenerateCacheKey(HttpRequest request, string idempotencyKey)
    {
        var raw = $"{request.Method}|{request.Path}|{idempotencyKey}";
        using var sha = SHA256.Create();
        var hash = sha.ComputeHash(Encoding.UTF8.GetBytes(raw));
        return Convert.ToBase64String(hash);
    }

    public async Task<byte[]> GetResponseAsync(string cacheKey)
    {
        return await _cache.GetAsync(cacheKey);
    }

    public async Task StoreResponseAsync(string cacheKey, byte[] serializedResponse, TimeSpan ttl)
    {
        await _cache.SetAsync(cacheKey, serializedResponse, new DistributedCacheEntryOptions
        {
            AbsoluteExpirationRelativeToNow = ttl
        });
    }
}
```

**Serialization:** Convert response object to JSON bytes before storing.

```csharp
var responseBytes = JsonSerializer.SerializeToUtf8Bytes(result);
await _redisIdempotency.StoreResponseAsync(cacheKey, responseBytes, TimeSpan.FromHours(24));
```

When retrieving, deserialize back to the expected type.

**Why hash the key?** Prevents specially crafted idempotency keys from causing collisions or exposing internal data. The hash ensures a fixed‑length, safe cache key.


## 7. Best Practices for RESTful API Design <a name="best-practices"></a>

1. **Use nouns, not verbs** – `/products` not `/getProducts`.
2. **Use plural resource names** – `/products` (collection), `/products/123` (single).
3. **Use HTTP methods correctly** – follow semantics.
4. **Return appropriate status codes** – 200, 201, 400, 404, etc.
5. **Provide consistent error responses** – standard structure.
6. **Version your API** – e.g., `/v1/products`.
7. **Support filtering, sorting, pagination** – query parameters.
8. **Use hypermedia (HATEOAS) optionally** – links to related resources.
9. **Secure your API** – HTTPS, authentication, rate limiting.
10. **Document thoroughly** – OpenAPI/Swagger.
11. **Implement idempotency for non‑safe methods** – use idempotency keys with cache.


## 8. Implementation Example in ASP.NET Core <a name="implementation"></a>

A complete RESTful controller for products (GET, POST, PUT, PATCH, DELETE) plus idempotent POST using in‑memory cache.

```csharp
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private static readonly List<Product> _products = new();
    private static int _nextId = 1;
    private readonly IMemoryCache _cache;

    public ProductsController(IMemoryCache cache) => _cache = cache;

    // GET /api/products
    [HttpGet]
    public IActionResult GetAll() => Ok(_products);

    // GET /api/products/{id}
    [HttpGet("{id}")]
    public IActionResult GetById(int id)
    {
        var product = _products.FirstOrDefault(p => p.Id == id);
        return product == null ? NotFound() : Ok(product);
    }

    // POST /api/products (idempotent)
    [HttpPost]
    public IActionResult Create([FromBody] Product product)
    {
        // Idempotency key header
        if (!Request.Headers.TryGetValue("Idempotency-Key", out var keyHeader))
            return BadRequest("Idempotency-Key header required");
        var idempotencyKey = keyHeader.ToString();

        // Generate cache key
        var raw = $"{Request.Method}|{Request.Path}|{idempotencyKey}";
        var hash = Convert.ToBase64String(SHA256.HashData(Encoding.UTF8.GetBytes(raw)));
        
        // Check cache
        if (_cache.TryGetValue(hash, out Product cachedProduct))
            return Ok(cachedProduct);

        // Create new product
        product.Id = _nextId++;
        _products.Add(product);

        // Store in cache (24 hours)
        _cache.Set(hash, product, TimeSpan.FromHours(24));

        return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
    }

    // PUT /api/products/{id}
    [HttpPut("{id}")]
    public IActionResult Replace(int id, Product updated)
    {
        var existing = _products.FirstOrDefault(p => p.Id == id);
        if (existing == null) return NotFound();
        existing.Name = updated.Name;
        existing.Price = updated.Price;
        return Ok(existing);
    }

    // DELETE /api/products/{id}
    [HttpDelete("{id}")]
    public IActionResult Delete(int id)
    {
        var product = _products.FirstOrDefault(p => p.Id == id);
        if (product == null) return NotFound();
        _products.Remove(product);
        return NoContent();
    }
}
```


## 9. Conclusion <a name="conclusion"></a>

RESTful APIs provide a scalable, stateless, and uniform way to expose resources over HTTP. Understanding HTTP methods, status codes, and idempotency is crucial for building reliable APIs. For non‑idempotent operations like `POST`, implement idempotency using an idempotency key header and a cache (in‑memory or Redis). Hash the key together with the HTTP method and request path to guarantee uniqueness across different endpoints.

By following the best practices outlined in this guide, you can design APIs that are robust, developer‑friendly, and safe to retry.
