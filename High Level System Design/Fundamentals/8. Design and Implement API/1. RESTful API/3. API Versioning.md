# Comprehensive Guide to API Versioning

API versioning allows you to introduce changes, add new features, or deprecate old endpoints without breaking existing clients. This document covers the most common strategies, their pros and cons, best practices, and implementation examples in ASP.NET Core.


## 1. Why API Versioning?

APIs evolve over time. Without versioning, a breaking change (renaming a field, changing a response structure, removing an endpoint) would break all existing clients. Versioning gives you a smooth transition path:

- Introduce new functionality without affecting existing clients.
- Deprecate old versions gradually.
- Maintain backward compatibility while innovating.

**Common breaking changes that require a new version:**
- Removing or renaming a property.
- Changing the data type of a field.
- Changing the request/response structure (e.g., nesting vs flat).
- Changing the behaviour of an endpoint (e.g., sorting defaults).
- Adding a new mandatory field.


## 2. API Versioning Strategies

There are four primary ways to specify the version in an API request. Each has its own trade-offs.

| Strategy               | Example                                      | Pros                                                                 | Cons                                                                 |
|------------------------|----------------------------------------------|----------------------------------------------------------------------|----------------------------------------------------------------------|
| **URL Path**           | `/api/v1/products`, `/api/v2/products`      | Most visible, easy to cache, simple to implement, RESTful            | Can lead to duplicate code if not managed well, clutters URLs        |
| **Query String**       | `/api/products?version=1`, `?version=2`     | Simple, doesn’t change URL structure                                 | Can be ignored by clients, not as self‑documenting, caching issues   |
| **Custom Header**      | `api-version: 1`                            | Keeps URLs clean, version is metadata, difficult to forget           | Not visible in browser/curl by default, harder to test manually      |
| **Media Type (Accept)**| `Accept: application/vnd.myapi.v2+json`     | Follows content negotiation, version is part of contract             | Complex to implement, client must set proper Accept header           |

### 2.1 URL Path Versioning (Most Popular)

```
GET /api/v1/products
GET /api/v2/products
```

**When to use:** Public APIs, long‑lived APIs, when you want the version to be obvious in every request.

**Implementation in ASP.NET Core:** Use attribute routing with a version parameter.

### 2.2 Query String Versioning

```
GET /api/products?api-version=1
GET /api/products?api-version=2
```

**When to use:** Internal or private APIs, when you want to keep URLs clean and simple.

**Implementation:** Use the `Microsoft.AspNetCore.Mvc.Versioning` package.

### 2.3 Header Versioning

```
GET /api/products
api-version: 1
```

**When to use:** When you want the URL to be version‑agnostic and force clients to explicitly set a header.

**Implementation:** Same package, configure to read from header.

### 2.4 Media Type Versioning (Accept Header)

```
GET /api/products
Accept: application/vnd.mycompany.v1+json
```

**When to use:** REST purists, when you want version to be part of the resource representation contract.

**Implementation:** Custom media type mapping with the versioning library.


## 3. Best Approaches and Recommendations

After evaluating many real‑world APIs (Stripe, GitHub, Microsoft Graph, AWS), the **URL path versioning** is the most widely adopted and recommended for most scenarios because:

- It’s explicit and self‑documenting.
- Easy to cache (CDNs, reverse proxies) because the URL changes.
- Simple to test (just change the number in the URL).
- Works with any HTTP client without special headers.

**However, you can combine strategies:** For example, support both URL path and header, but choose one as the primary.

### Best Practices Summary

1. **Use integer versions** (v1, v2) – avoid semantic versioning (v1.2.3) in URLs.
2. **Keep versions simple** – major version only; minor changes should be backward compatible.
3. **Document version lifecycle** – clearly state support timelines for each version.
4. **Deprecate gracefully** – return `Deprecation` header or `Sunset` header (RFC 8594).
5. **Version the entire API, not individual endpoints** – all endpoints under `/api/v1/` share the same version.
6. **Avoid breaking changes within a version** – add new fields, don’t remove or rename.
7. **Use a versioning library** – don’t reinvent the wheel. `Microsoft.AspNetCore.Mvc.Versioning` is excellent.
8. **Set a default version** – for clients that don’t specify one.
9. **Return version info in responses** – e.g., a custom header `api-supported-versions`.
10. **Test all versions** – automate regression tests.


## 4. Implementing API Versioning in ASP.NET Core

We’ll use the official package: **`Microsoft.AspNetCore.Mvc.Versioning`**.

### 4.1 Installation

```bash
dotnet add package Microsoft.AspNetCore.Mvc.Versioning
```

Optionally, for API Explorer (Swagger) support:
```bash
dotnet add package Microsoft.AspNetCore.Mvc.Versioning.ApiExplorer
```

### 4.2 Basic Configuration (Program.cs)

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddApiVersioning(options =>
{
    // Default version when client doesn't specify
    options.DefaultApiVersion = new ApiVersion(1, 0);
    // Assume default version for unversioned requests
    options.AssumeDefaultVersionWhenUnspecified = true;
    // Report supported versions in response headers
    options.ReportApiVersions = true;
    
    // Choose versioning strategy (can combine multiple)
    options.ApiVersionReader = ApiVersionReader.Combine(
        new UrlSegmentApiVersionReader(),        // v1, v2 in route
        new QueryStringApiVersionReader("api-version"),
        new HeaderApiVersionReader("api-version"),
        new MediaTypeApiVersionReader()          // from Accept header
    );
});

// Add API Explorer for Swagger (optional)
builder.Services.AddVersionedApiExplorer(options =>
{
    options.GroupNameFormat = "'v'VVV";
    options.SubstituteApiVersionInUrl = true;
});

builder.Services.AddControllers();
```

### 4.3 URL Path Versioning Example

Use `[Route("api/v{version:apiVersion}/[controller]")]`.

```csharp
[ApiController]
[Route("api/v{version:apiVersion}/[controller]")]
[ApiVersion("1.0")]
public class ProductsController : ControllerBase
{
    [HttpGet]
    public IActionResult GetV1() => Ok(new[] { "Product A (v1)", "Product B (v1)" });
}

[ApiController]
[Route("api/v{version:apiVersion}/[controller]")]
[ApiVersion("2.0")]
public class ProductsController : ControllerBase
{
    [HttpGet]
    public IActionResult GetV2() => Ok(new[] { 
        new { Name = "Product X", Price = 100 }, 
        new { Name = "Product Y", Price = 200 } 
    });
}
```

**Requests:**
- `GET /api/v1/products` → v1 response.
- `GET /api/v2/products` → v2 response.

### 4.4 Query String Versioning

Same controller can support multiple versions by decorating with `[ApiVersion]` attributes.

```csharp
[ApiController]
[Route("api/products")]
[ApiVersion("1.0")]
[ApiVersion("2.0")]
public class ProductsController : ControllerBase
{
    [MapToApiVersion("1.0")]
    [HttpGet]
    public IActionResult GetV1() => Ok("Version 1");

    [MapToApiVersion("2.0")]
    [HttpGet]
    public IActionResult GetV2() => Ok("Version 2");
}
```

Now clients call: `GET /api/products?api-version=1.0` or `GET /api/products?api-version=2.0`.

### 4.5 Header Versioning

Same controller, but version is read from `api-version` header.

Configuration:
```csharp
options.ApiVersionReader = new HeaderApiVersionReader("api-version");
```

### 4.6 Combined Versioning (URL + Header + Query)

Use `ApiVersionReader.Combine()` as shown in the basic config. The library will try each reader in order.

### 4.7 Deprecating a Version

```csharp
[ApiController]
[Route("api/v{version:apiVersion}/products")]
[ApiVersion("1.0", Deprecated = true)]
public class ProductsV1Controller : ControllerBase { ... }
```

The `ReportApiVersions` option will add a header like:
```
api-supported-versions: 1.0, 2.0
api-deprecated-versions: 1.0
```

### 4.8 Mapping to Different Controller Classes per Version

For clarity, you can separate versions into different namespaces and controllers.

```csharp
// V1 controller
namespace MyApi.Controllers.V1
{
    [ApiController]
    [Route("api/v{version:apiVersion}/products")]
    [ApiVersion("1.0")]
    public class ProductsController : ControllerBase { ... }
}

// V2 controller
namespace MyApi.Controllers.V2
{
    [ApiController]
    [Route("api/v{version:apiVersion}/products")]
    [ApiVersion("2.0")]
    public class ProductsController : ControllerBase { ... }
}
```

### 4.9 Swagger/OpenAPI Integration with Multiple Versions

Install `Microsoft.AspNetCore.Mvc.Versioning.ApiExplorer`. Then configure Swagger:

```csharp
builder.Services.AddSwaggerGen();
builder.Services.ConfigureOptions<ConfigureSwaggerOptions>();

// Helper class to generate Swagger docs per version
public class ConfigureSwaggerOptions : IConfigureOptions<SwaggerGenOptions>
{
    private readonly IApiVersionDescriptionProvider _provider;
    public ConfigureSwaggerOptions(IApiVersionDescriptionProvider provider) => _provider = provider;

    public void Configure(SwaggerGenOptions options)
    {
        foreach (var description in _provider.ApiVersionDescriptions)
        {
            options.SwaggerDoc(description.GroupName, new OpenApiInfo
            {
                Title = $"API {description.ApiVersion}",
                Version = description.ApiVersion.ToString(),
                Description = description.IsDeprecated ? "Deprecated" : ""
            });
        }
    }
}

// In app
app.UseSwagger();
app.UseSwaggerUI(options =>
{
    foreach (var desc in app.Services.GetRequiredService<IApiVersionDescriptionProvider>().ApiVersionDescriptions)
    {
        options.SwaggerEndpoint($"/swagger/{desc.GroupName}/swagger.json", desc.GroupName.ToUpperInvariant());
    }
});
```


## 5. Advanced Scenarios

### 5.1 Versioning a Single Action (Not Whole Controller)

```csharp
[ApiController]
[Route("api/products")]
[ApiVersionNeutral] // controller is version-neutral
public class ProductsController : ControllerBase
{
    [MapToApiVersion("1.0")]
    [HttpGet("legacy")]
    public IActionResult Legacy() => Ok("Legacy");

    [MapToApiVersion("2.0")]
    [HttpGet("latest")]
    public IActionResult Latest() => Ok("Latest");
}
```

### 5.2 Custom Version Reader

Implement `IApiVersionReader` to read version from, e.g., a JWT claim or custom header.

### 5.3 Versioning via URL Subdomain

```
https://v1.api.mycompany.com/products
https://v2.api.mycompany.com/products
```

This can be done by mapping different subdomains to different versions of the app (separate deployments) or using rewrite middleware.

### 5.4 Internal vs External Versioning

You might use internal versions for developers (e.g., `/api/v1`) but external versions for partners (e.g., `/api/2025-03`). Choose a scheme that matches your audience.

## 7. Versioning Strategies Comparison Table

| Strategy       | Cache Friendly | RESTful | Ease of Implementation | Client Effort | Self-Documenting |
|----------------|----------------|---------|------------------------|---------------|------------------|
| URL Path       | ✅ Excellent    | ✅ Yes  | ✅ Easy                | Low (just change URL) | ✅ Yes |
| Query String   | ⚠️ Moderate     | ⚠️ Debatable | ✅ Easy            | Low           | ❌ No (easy to miss) |
| Header         | ✅ Good (URL same) | ⚠️ Debatable | ⚠️ Medium        | Medium (set header) | ❌ No |
| Media Type     | ✅ Good         | ✅ Yes (RESTful) | ❌ Complex      | High (custom Accept) | ❌ No (not visible) |


## 8. Real‑World Examples

- **GitHub API:** URL path (`/api/v3/users`). They rarely change version but add new fields.
- **Stripe API:** Uses a custom `Stripe-Version` header. They also version the API behavior.
- **Microsoft Graph:** Uses URL path (`/v1.0/me`, `/beta/me`).
- **AWS API:** Uses query string (`?Version=2012-08-10`).


## 9. Complete Example: API with URL Path Versioning

**Program.cs**
```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddApiVersioning(options =>
{
    options.DefaultApiVersion = new ApiVersion(1, 0);
    options.AssumeDefaultVersionWhenUnspecified = true;
    options.ReportApiVersions = true;
    options.ApiVersionReader = new UrlSegmentApiVersionReader();
});
builder.Services.AddVersionedApiExplorer(options =>
{
    options.GroupNameFormat = "'v'VVV";
    options.SubstituteApiVersionInUrl = true;
});
builder.Services.AddControllers();
builder.Services.AddSwaggerGen();
builder.Services.ConfigureOptions<ConfigureSwaggerOptions>();

var app = builder.Build();
app.UseSwagger();
app.UseSwaggerUI(c =>
{
    var provider = app.Services.GetRequiredService<IApiVersionDescriptionProvider>();
    foreach (var desc in provider.ApiVersionDescriptions)
        c.SwaggerEndpoint($"/swagger/{desc.GroupName}/swagger.json", desc.GroupName);
});
app.MapControllers();
app.Run();
```


## 10. Summary and Final Recommendations

- **For most public REST APIs:** Use **URL path versioning** (`/v1/...`, `/v2/...`). It’s the clearest and easiest for clients.
- **For internal or B2B APIs:** Query string or header versioning can keep URLs clean.
- **Always use a versioning library** – don’t parse versions manually.
- **Document your versioning policy** and deprecation schedule.
- **Return version info in headers** so clients know what’s available.
- **Use API Explorer to generate Swagger docs per version** – this is a huge help for consumers.
- **Never change the behaviour of an existing version** – create a new version instead.
- **Plan for sunset** – give clients ample warning (6–12 months) before removing an old version.
