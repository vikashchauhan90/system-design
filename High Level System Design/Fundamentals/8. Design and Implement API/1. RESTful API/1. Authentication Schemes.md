# Comprehensive Guide to Authentication Schemes

Below is a detailed breakdown of common authentication mechanisms, their pros/cons, and C# implementation examples.


## 1. Basic Authentication

**Description**  
Sends username and password in plaintext Base64-encoded via the `Authorization: Basic <credentials>` header. Usually combined with HTTPS.

**Pros**  
- Extremely simple to implement  
- Supported by virtually all HTTP clients  
- No session state on server  

**Cons**  
- Credentials sent with every request (increases exposure)  
- Must always use HTTPS, otherwise credentials are easily decoded  
- No built-in logout mechanism  
- Vulnerable to CSRF if not protected  

**Use case**  
Simple internal APIs, legacy systems, or quick prototypes where HTTPS is enforced.

**Tips**  
- **Always use HTTPS** – credentials are only Base64 encoded, not encrypted.  
- Consider rate‑limiting to prevent brute‑force attacks.  
- Avoid for public or user‑facing applications.

**C# Example (ASP.NET Core)**  
```csharp
app.Use(async (context, next) =>
{
    if (context.Request.Headers.ContainsKey("Authorization"))
    {
        var authHeader = context.Request.Headers["Authorization"].ToString();
        if (authHeader.StartsWith("Basic ", StringComparison.OrdinalIgnoreCase))
        {
            var encoded = authHeader.Substring("Basic ".Length).Trim();
            var decoded = Encoding.UTF8.GetString(Convert.FromBase64String(encoded));
            var parts = decoded.Split(':', 2);
            var username = parts[0];
            var password = parts[1];

            if (username == "admin" && password == "secret")
                context.Items["User"] = username;
        }
    }
    await next();
});
```

## 2. Bearer JWT Token

**Description**  
JSON Web Token (JWT) containing claims, signed by the server. Client sends it in `Authorization: Bearer <token>` header.

**Pros**  
- Stateless – no server-side storage  
- Self-contained (user info, roles, expiration)  
- Can be used across multiple services (SSO)  
- Supports encryption and signing  

**Cons**  
- Cannot be revoked easily without a blacklist  
- Token size can be large  
- Secret key management is critical  
- No built-in way to refresh token (requires refresh token mechanism)

**Use case**  
Stateless APIs, microservices, mobile apps, single‑page applications (SPAs).

**Tips**  
- **Always validate the token signature** with the correct signing key – never trust a JWT without verification.  
- Keep secret keys secure (use Azure Key Vault, AWS KMS, or environment variables).  
- Set short expiration times (15–60 minutes) and use refresh tokens.  
- Store tokens in memory (SPA) or secure storage (mobile), never in `localStorage` if XSS is a concern.

**C# Example (ASP.NET Core)**  
```csharp
// Configure JWT authentication
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidIssuer = "your-issuer",
            ValidAudience = "your-audience",
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes("your-secret-key"))
        };
    });

// Issuing a token
var tokenHandler = new JwtSecurityTokenHandler();
var key = Encoding.ASCII.GetBytes("your-secret-key");
var tokenDescriptor = new SecurityTokenDescriptor
{
    Subject = new ClaimsIdentity(new[] { new Claim(ClaimTypes.Name, "username") }),
    Expires = DateTime.UtcNow.AddHours(1),
    SigningCredentials = new SigningCredentials(new SymmetricSecurityKey(key), SecurityAlgorithms.HmacSha256Signature)
};
var token = tokenHandler.CreateToken(tokenDescriptor);
var tokenString = tokenHandler.WriteToken(token);
```

## 3. Session-Based Authentication

**Description**  
Server creates a session (stored in memory, database, or cache) after login, and sends a session ID to the client (usually via cookie). Subsequent requests include the session ID.

**Pros**  
- Easy to revoke (delete session on server)  
- Can store large amounts of user data server-side  
- Works with any client that supports cookies  

**Cons**  
- Stateful – consumes server memory/resources  
- Difficult to scale horizontally (needs shared session storage)  
- Vulnerable to CSRF if not protected with anti-forgery tokens  

**Use case**  
Traditional server‑rendered web apps (Razor Pages, MVC) where you need easy revocation and server‑side state.

**Tips**  
- **Use the same domain** for all applications that share the session cookie (to prevent cookie fragmentation).  
- Store session data in a distributed cache (Redis, SQL Server) for horizontal scaling.  
- Set `HttpOnly` and `Secure` flags on the session cookie.  
- Implement anti‑CSRF tokens.

**C# Example (ASP.NET Core)**  
```csharp
builder.Services.AddDistributedMemoryCache(); // or Redis/SQL Server
builder.Services.AddSession(options =>
{
    options.IdleTimeout = TimeSpan.FromMinutes(30);
    options.Cookie.HttpOnly = true;
});

app.UseSession();

// Login
HttpContext.Session.SetString("UserId", "123");
HttpContext.Session.SetInt32("Role", 1);

// Check session
if (HttpContext.Session.GetString("UserId") != null) { /* authenticated */ }
```

## 4. Cookie-Based Authentication

**Description**  
Similar to session but the authentication ticket itself (encrypted) is stored in the cookie. Often used with ASP.NET Core Identity.

**Pros**  
- Stateless on server (cookie contains all data)  
- Automatic handling by browsers  
- Can be encrypted and signed  

**Cons**  
- Cookie size limits (4KB)  
- Vulnerable to CSRF and XSS (if not marked HttpOnly/Secure)  
- Cannot be easily revoked unless using a server-side revocation list  

**Use case**  
Small web apps where you want to avoid server‑side storage but still use cookies.

**Tips**  
- **Same domain rule** – cookies are bound to a domain; subdomains can share if domain attribute is set correctly.  
- Encrypt and sign the cookie to prevent tampering.  
- Limit cookie size (<4KB).  
- Use `SameSite=Strict` or `Lax` to mitigate CSRF.

**C# Example (ASP.NET Core Identity)**  
```csharp
builder.Services.AddAuthentication(CookieAuthenticationDefaults.AuthenticationScheme)
    .AddCookie(options =>
    {
        options.LoginPath = "/Account/Login";
        options.Cookie.HttpOnly = true;
        options.Cookie.SameSite = SameSiteMode.Strict;
        options.Cookie.SecurePolicy = CookieSecurePolicy.Always;
        options.ExpireTimeSpan = TimeSpan.FromDays(7);
    });

// Sign in
await HttpContext.SignInAsync("username", new ClaimsPrincipal(identity));
```

## 5. Opaque Token

**Description**  
A reference token that has no intrinsic meaning; it’s a random identifier that the client presents, and the server must look up the associated data in a store (e.g., database, Redis).

**Pros**  
- Easy to revoke instantly (delete from store)  
- Small token size (just an ID)  
- Can store arbitrary data server-side  

**Cons**  
- Stateful – requires server-side storage  
- Every request hits the token store (latency)  
- Harder to scale without distributed cache  

**Use case**  
High‑security environments where immediate revocation is mandatory (e.g., financial APIs).

**Tips**  
- Store tokens in a fast key‑value store (Redis) with expiration.  
- Each request requires a lookup – use caching to reduce latency.  
- Token should be a random, unguessable string (e.g., `Guid.NewGuid()` + random suffix).

**C# Example (Custom)**  
```csharp
// Generate opaque token (random GUID)
var token = Guid.NewGuid().ToString();
await db.Tokens.AddAsync(new Token { Id = token, UserId = "123", Expires = DateTime.UtcNow.AddHours(1) });
await db.SaveChangesAsync();

// Validate token
var tokenRecord = await db.Tokens.FindAsync(token);
if (tokenRecord != null && tokenRecord.Expires > DateTime.UtcNow)
{
    // authenticate user
}
```

## 6. Two-Factor Authentication (2FA)

**Description**  
Requires an additional factor beyond password, e.g., TOTP (authenticator app), SMS, or email code.

**Pros**  
- Significantly improves security  
- Protects against credential theft  
- Widely supported (Google Authenticator, Microsoft Authenticator)  

**Cons**  
- Adds friction for users  
- SMS-based 2FA is vulnerable to SIM swapping  
- Requires backup codes for account recovery  

**Use case**  
Adding an extra security layer for sensitive operations (login, payment, profile changes).

**Tips**  
- Prefer TOTP (authenticator apps) over SMS – SMS is vulnerable to SIM swapping.  
- Provide backup codes for account recovery.  
- Remember trusted devices for limited time to reduce friction.

**C# Example (TOTP with ASP.NET Core Identity)**  
```csharp
// Generate secret key
var secretKey = KeyGeneration.GenerateNewKey();
var qrCodeUrl = OtpNet.QrCodeHelper.GenerateQrCodeUri("MyApp", user.Email, secretKey);

// Verify TOTP
var totp = new Totp(secretKey);
bool isValid = totp.VerifyTotp(userCode, out _);
```

## 7. OAuth 2.0 (Authorization only)

> **Note**: OAuth 2.0 alone does **not** provide authentication. It gives an access token (e.g., to access a user’s Google Drive) without telling you *who* the user is.

**Description**  
OAuth 2.0 is an authorization framework that allows a third-party application to obtain limited access to an HTTP service on behalf of the resource owner.

**Pros**  
- Delegated access without sharing credentials  
- Fine‑grained scopes and permissions  
- Widely adopted by major platforms (Google, Facebook, Microsoft)  

**Cons**  
- Does not provide user identity (only access token)  
- Complex to implement correctly (multiple flows)  
- Requires secure handling of client secrets  

**Use case**  
Delegated access – “Let this app read my calendar” (without giving away password).

**Tips**  
- Always use the **Authorization Code Flow** with PKCE for public clients (SPA, mobile).  
- Validate the redirect URI to prevent open redirect attacks.  
- Store client secrets securely for confidential clients.

**C# Example (using IdentityModel)**  
```csharp
var client = new HttpClient();
var response = await client.RequestAuthorizationCodeTokenAsync(new AuthorizationCodeTokenRequest
{
    Address = tokenEndpoint,
    ClientId = "client_id",
    ClientSecret = "client_secret",
    Code = authorizationCode,
    RedirectUri = redirectUri
});
var accessToken = response.AccessToken;
```

## 8. OpenID Connect (OIDC) – Authentication + Identity

**Description**  
OpenID Connect is an identity layer on top of OAuth 2.0. It provides authentication and returns an ID token (JWT) containing user identity claims.

**Pros**  
- Delegated authentication – no need to store passwords  
- Supports social logins and enterprise federation  
- Provides ID token (JWT) with user claims  
- Built-in support for logout, session management  

**Cons**  
- External dependency (identity provider)  
- Can be costly at scale  
- Requires understanding of OAuth 2.0 flows (authorization code, PKCE, etc.)  

**Use case**  
Single Sign‑On (SSO) to multiple applications, social login (Google, Facebook), enterprise identity federation.

**Tips**  
- **OIDC = OAuth 2.0 + ID Token** – the ID token (a JWT) contains identity claims (sub, email, name).  
- Validate the `iss` (issuer) and `aud` (audience) of the ID token.  
- Use the `UserInfo` endpoint if you need additional claims.  
- Always use `state` parameter to prevent CSRF.

**C# Example (Auth0 with ASP.NET Core)**  
```csharp
builder.Services.AddAuthentication(options =>
{
    options.DefaultAuthenticateScheme = CookieAuthenticationDefaults.AuthenticationScheme;
    options.DefaultChallengeScheme = OpenIdConnectDefaults.AuthenticationScheme;
})
.AddCookie()
.AddOpenIdConnect(options =>
{
    options.Authority = $"https://{yourDomain}.auth0.com";
    options.ClientId = "your-client-id";
    options.ClientSecret = "your-client-secret";
    options.ResponseType = "code";
    options.Scope.Add("openid");
    options.Scope.Add("profile");
    options.Scope.Add("email");
    options.CallbackPath = "/signin-auth0";
    options.SignedOutCallbackPath = "/signout-callback-auth0";
});
```

## 9. Single Sign-On (SSO)

**Description**  
Allows users to log in once and access multiple independent applications. Common protocols: SAML, OAuth2/OIDC, Kerberos.

**Pros**  
- Improves user experience (no multiple logins)  
- Centralized user management and security policies  
- Reduces password fatigue  

**Cons**  
- Complex to implement (requires federation)  
- Single point of failure (identity provider)  
- Requires trust between all participating apps  

**Use case**  
Multiple applications under the same organization (or federated) – users log in once and access all.

**Tips**  
- Use a **federation protocol** – SAML (enterprise) or OIDC (modern).  
- Centralise identity management (Azure AD, Okta, Auth0).  
- Plan for logout (single logout – SLO) to clear session across all apps.

**C# Example (SAML 2.0 with ITfoxtec Identity SAML2)**  
```csharp
// Configure SAML2
app.UseSaml2(saml2Options =>
{
    saml2Options.SPOptions.EntityId = new EntityId("your-sp-entity-id");
    saml2Options.IdentityProviders.Add(new IdentityProvider(
        new EntityId("idp-entity-id"), saml2Options.SPOptions)
    {
        SingleSignOnDestination = new Uri("https://idp/sso"),
        SingleLogoutDestination = new Uri("https://idp/slo"),
        MetadataLocation = "https://idp/metadata.xml",
    });
});

// In login action
var saml2AuthnRequest = new Saml2AuthnRequest(saml2Options);
return saml2AuthnRequest.Bind(new HttpRequestData(HttpContext.Request));
```

## 10. Windows Authentication (Kerberos / NTLM)

**Description**  
Uses Windows Active Directory to authenticate users via Kerberos (preferred) or NTLM (fallback). The web server (IIS) or Kestrel with HTTP.sys negotiates the authentication handshake.

**Pros**  
- Seamless SSO experience for domain users  
- Centralized identity management via Active Directory  
- No need for users to manage separate credentials  

**Cons**  
- Ties the application to Windows and Active Directory  
- More complex to configure for cross‑platform or cloud‑native environments  
- Not suitable for public internet applications  

**Use case**  
Intranet web applications within a corporate network; internal Line‑of‑Business (LoB) apps.

**Tips**  
- **Use Kerberos over NTLM** – Kerberos is more secure.  
- **Ensure HTTPS** – prevents relay attacks and encrypts traffic.  
- Browser must support Negotiate/Kerberos (standard in domain‑joined environments).  
- Avoid for public‑facing or cross‑platform apps.

**C# Example (ASP.NET Core with HTTP.sys)**  
```csharp
builder.Services.AddAuthentication(NegotiateDefaults.AuthenticationScheme)
   .AddNegotiate();

builder.Services.AddAuthorization();
app.UseAuthentication();
app.UseAuthorization();
```

## 11. Magic Link (Passwordless Authentication)

**Description**  
A one‑time‑use URL sent to the user’s email address. Clicking the link authenticates the user and creates a session.

**Pros**  
- Eliminates password‑related risks (weak passwords, reuse, phishing)  
- Simple, user‑friendly login experience  
- Reduces support costs for password resets  

**Cons**  
- Relies on the security of the user’s email account  
- Email delivery delays can hinder user experience  
- Susceptible to email interception if the connection is not secure  

**Use case**  
Consumer applications where low‑friction sign‑in is desired; as a second factor in MFA.

**Tips**  
- **Short expiry** – tokens should expire within 5‑15 minutes.  
- **One‑time use** – invalidate token immediately after use.  
- **Secure callback** – validate the callback URL to prevent open redirects.  
- **Secure email** – encourage users to enable 2FA on their email accounts.

**C# Example (Custom token provider)**  
```csharp
public class PasswordlessLoginProvider<TUser> : IUserTwoFactorTokenProvider<TUser> where TUser : class
{
    public async Task<string> GenerateAsync(string purpose, UserManager<TUser> manager, TUser user)
    {
        var token = await manager.GenerateUserTokenAsync(user, TokenOptions.DefaultProvider, purpose);
        // Build magic link: https://yourapp.com/login/callback?userId={user.Id}&token={token}
        return token;
    }

    public async Task<bool> ValidateAsync(string purpose, string token, UserManager<TUser> manager, TUser user)
    {
        return await manager.VerifyUserTokenAsync(user, TokenOptions.DefaultProvider, purpose, token);
    }
}
```

## 12. Mutual TLS (mTLS)

**Description**  
Both client and server authenticate each other using X.509 digital certificates. Ensures both parties are who they claim to be.

**Pros**  
- Highest level of security for server‑to‑server communication  
- Strong, non‑repudiable identity verification  
- Ideal for zero‑trust network models  

**Cons**  
- Complex to set up and manage the PKI infrastructure  
- Performance overhead due to cryptographic operations  
- Challenging to use in standard web browsers  

**Use case**  
Secure communication between backend microservices; authenticating IoT devices; machine‑to‑machine (M2M) APIs.

**Tips**  
- **Use a robust PKI** – manage certificates with a trusted Certificate Authority (CA).  
- **Short‑lived certificates** – automate renewal to reduce impact of compromise.  
- **Combine with authorization** – mTLS verifies identity; still need permissions.  
- Avoid self‑signed certificates in production.

**C# Example (ASP.NET Core Kestrel)**  
```csharp
builder.WebHost.ConfigureKestrel(serverOptions =>
{
    serverOptions.ConfigureHttpsDefaults(listenOptions =>
    {
        listenOptions.ClientCertificateMode = ClientCertificateMode.RequireCertificate;
        listenOptions.ClientCertificateValidation = (cert, chain, errors) =>
        {
            // Custom validation logic (e.g., thumbprint, issuer)
            return true;
        };
    });
});
```

## 13. HMAC (Hash‑Based Message Authentication Code)

**Description**  
Client creates a signature of the request (method, URL, body, timestamp) using a shared secret key. Server recalculates and compares the signature to verify authenticity and integrity.

**Pros**  
- Stateless and efficient to compute  
- Provides request integrity and non‑repudiation  
- Widely used in API security (e.g., AWS Signature V4)  

**Cons**  
- Relies on secure distribution of the shared secret  
- Does not encrypt the request payload (use TLS)  
- Complex to implement correctly (canonical request, replay protection)  

**Use case**  
Securing public APIs for partners; webhook security; lightweight service‑to‑service authentication.

**Tips**  
- **Always use TLS** – HMAC provides integrity, not confidentiality.  
- **Use strong secrets** – long, random keys stored in a vault.  
- **Include a timestamp** – prevent replay attacks (accept only a small window).  
- **Use a strong hash** – prefer SHA‑256 or SHA‑3.

**C# Example (Custom AuthenticationHandler)**  
```csharp
public class HmacAuthenticationHandler : AuthenticationHandler<HmacAuthenticationOptions>
{
    protected override async Task<AuthenticateResult> HandleAuthenticateAsync()
    {
        if (!Request.Headers.TryGetValue("Authorization", out var authHeader))
            return AuthenticateResult.NoResult();

        // Expected format: "HMAC <keyId>:<signature>"
        var parts = authHeader.ToString().Split(' ', 2);
        if (parts.Length != 2 || parts[0] != "HMAC")
            return AuthenticateResult.Fail("Invalid format");

        var keyAndSignature = parts[1].Split(':');
        var clientId = keyAndSignature[0];
        var clientSignature = keyAndSignature[1];
```

## 14. PKCE (Proof Key for Code Exchange)

**Description**  
An OAuth 2.0 extension that protects the authorization code flow from interception attacks, especially for public clients (mobile apps, SPAs) that cannot store a client secret.

**Pros**  
- Prevents authorization code interception attacks  
- Eliminates the need for a client secret in public clients  
- Mandated by many modern OAuth 2.0/OIDC security standards  

**Cons**  
- Adds extra steps to the OAuth flow  
- Not needed for confidential clients (web apps) that can securely store a secret  
- Increases client‑side complexity  

**Use case**  
Mobile apps (iOS, Android, UWP); Single‑Page Applications (SPAs); any public OAuth 2.0 client.

**Tips**  
- **PKCE is now required** by most identity platforms (Microsoft Entra ID, Auth0).  
- **Use a strong verifier** – high‑entropy, cryptographically random (min 43 characters).  
- **Use SHA‑256** challenge method (`S256`), not the plain method.  
- **Combine with `state` parameter** – PKCE protects the code exchange, `state` prevents CSRF.  
- **PKCE vs. nonce** – in OIDC, use `nonce` for ID token and PKCE for the code exchange; they are complementary.

**C# Example (using Kavez.PKCE library)**  
```csharp
using Kavez.PKCE;

public class PkceService
{
    public (string codeVerifier, string codeChallenge) GeneratePkcePair()
    {
        var codeVerifier = PkceGenerator.GenerateCodeVerifier();
        var codeChallenge = PkceGenerator.GenerateCodeChallenge(codeVerifier);
        return (codeVerifier, codeChallenge);
    }
}
```

## Final Notes

- **Never mix up OAuth and OpenID Connect** – OAuth is for *authorization*, OIDC for *authentication*.  
- **Always validate** – JWT signatures, redirect URIs, state parameters, and token expiration.  
- **Cookie domain** – use a common parent domain (e.g., `.example.com`) for shared authentication across subdomains.  
- **JWT signing key** – rotate keys periodically and use asymmetric keys (RS256) for distributed validation.  
- **PKCE** is not part of OIDC by default – it's an OAuth 2.0 extension, but OIDC providers often require it.  
- **HMAC** is not part of OAuth/OIDC – it's a separate request‑signing mechanism.

This guide covers the most widely used authentication schemes in modern applications. Choose the one that best fits your security requirements, user experience, and infrastructure constraints.
