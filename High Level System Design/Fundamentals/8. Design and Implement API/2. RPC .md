# Comprehensive Guide to RPC (Remote Procedure Call)


## 1. What is RPC?

**Remote Procedure Call (RPC)** is a protocol that allows a program to execute a procedure (function) on a remote server as if it were a local call. The client calls a function with parameters; the RPC framework packages (marshals) the request, sends it over the network, and the server executes the function and returns the result.

RPC abstracts the network layer, making distributed computing feel like local function invocation.

---

## 2. How RPC Works

```
Client                                Server
   |                                    |
   | 1. Call local stub                 |
   |----------------------------------->|
   | 2. Marshal arguments               |
   | 3. Send request over network       |
   |                                    |
   |                                    | 4. Unmarshal arguments
   |                                    | 5. Execute actual procedure
   |                                    | 6. Marshal result
   | 7. Receive response                |
   |<-----------------------------------|
   | 8. Unmarshal result                |
   | 9. Return to caller                |
```

**Key components:**
- **Stub (client‑side proxy):** Simulates the remote function locally.
- **Skeleton (server‑side dispatcher):** Receives requests, calls the actual implementation, and returns the response.
- **Marshalling:** Converting data structures into a wire format (binary or text).
- **Transport:** Usually HTTP/2 (gRPC), TCP, or HTTP/1.1.

---

## 3. Types of RPC

### 3.1 gRPC (Google RPC)
- **Protocol:** HTTP/2 with Protocol Buffers (Protobuf) as the Interface Definition Language (IDL).
- **Features:** Bi‑directional streaming, multiplexing, deadlines, cancellation, pluggable auth.
- **Performance:** Very high (binary serialisation, HTTP/2).

### 3.2 JSON‑RPC
- **Protocol:** JSON over HTTP or TCP.
- **Features:** Simple, human‑readable, stateless, supports batching.
- **Performance:** Moderate (text‑based).

### 3.3 XML‑RPC
- **Protocol:** XML over HTTP.
- **Features:** Legacy, simple, but verbose.
- **Performance:** Low.

### 3.4 .NET Remoting / WCF (Legacy)
- **Protocol:** Binary or SOAP over TCP/HTTP.
- **Status:** Obsolete in modern .NET (use gRPC or ASP.NET Core Web API).

---

## 4. Pros and Cons of RPC

| Pros | Cons |
|------|------|
| **Simple programming model** – calls look like local functions. | **Tight coupling** – client and server share the same function signature. |
| **High performance** (especially gRPC with binary serialisation). | **Harder to evolve** – changing a contract may break clients. |
| **Strong typing** via IDL (Protocol Buffers). | **Not human‑readable** (binary protocols). |
| **Efficient for internal microservices** (low latency). | **Less browser‑friendly** (no direct HTML/JS support without gRPC‑web). |
| **Built‑in streaming** (gRPC). | **Debugging** – less transparent than REST. |
| **Code generation** across many languages. | **Complex tooling** compared to REST + Swagger. |

---

## 5. Use Cases

| Use Case | Recommended RPC Type |
|----------|----------------------|
| **Microservices communication** (high throughput, low latency) | gRPC |
| **Real‑time streaming** (e.g., chat, live data feeds) | gRPC (bidirectional streaming) |
| **Cross‑language services** (polyglot environments) | gRPC (with Protobuf) |
| **Simple internal services** (small teams, fast iteration) | JSON‑RPC |
| **IoT device communication** (lightweight, binary) | gRPC or custom TCP RPC |
| **Browser‑to‑backend (REST alternative)** | gRPC‑web or JSON‑RPC |
| **Legacy system integration** (XML‑based) | XML‑RPC (avoid for new projects) |

---

## 6. RPC vs REST vs GraphQL

| Feature | RPC (gRPC) | REST | GraphQL |
|---------|------------|------|---------|
| **Communication style** | Function call | Resource‑oriented | Query/mutation |
| **Protocol** | HTTP/2, TCP | HTTP/1.1, HTTP/2 | HTTP/1.1, HTTP/2 |
| **Data format** | Binary (Protobuf) | JSON, XML, etc. | JSON |
| **Performance** | Very high | Moderate | Moderate |
| **Browser support** | Via gRPC‑web | Native | Native |
| **Code generation** | Strong (many languages) | Optional (OpenAPI) | Optional (GraphQL codegen) |
| **Learning curve** | Moderate | Low | Moderate |
| **Best for** | Internal microservices | Public APIs, web | Complex queries, mobile |

## 7. C# Implementation Examples

### 7.1 gRPC with ASP.NET Core

**Prerequisites:** Install `Grpc.AspNetCore` package.

**Define the service in `.proto` file (`greet.proto`)**
```protobuf
syntax = "proto3";

service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest {
  string name = 1;
}

message HelloReply {
  string message = 1;
}
```

**Server Implementation (ASP.NET Core)**
```csharp
using Grpc.Core;

public class GreeterService : Greeter.GreeterBase
{
    public override Task<HelloReply> SayHello(HelloRequest request, ServerCallContext context)
    {
        return Task.FromResult(new HelloReply { Message = "Hello " + request.Name });
    }
}

// Program.cs
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddGrpc();
var app = builder.Build();
app.MapGrpcService<GreeterService>();
app.MapGet("/", () => "gRPC server running");
app.Run();
```

**Client Console App**
```csharp
using Grpc.Net.Client;
using var channel = GrpcChannel.ForAddress("https://localhost:5001");
var client = new Greeter.GreeterClient(channel);
var reply = await client.SayHelloAsync(new HelloRequest { Name = "World" });
Console.WriteLine(reply.Message);
```

**Pros:** Ultra‑fast, strongly typed, streaming.  
**Cons:** Requires `.proto` files; HTTP/2 only; limited browser support.

### 7.2 JSON‑RPC with JSON-RPC.NET

**Install:** `dotnet add package Buildalyzer.Workspaces`? Actually use `Microsoft.AspNetCore.JsonRpc` or `StreamJsonRpc`. For simplicity, we use `StreamJsonRpc` from Microsoft.

**Server (ASP.NET Core Middleware)**
```csharp
using StreamJsonRpc;

[JsonRpcService]
public class CalculatorService
{
    [JsonRpcMethod("add")]
    public int Add(int a, int b) => a + b;
}

// Program.cs
app.UseWebSockets();
app.Use(async (context, next) =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        var webSocket = await context.WebSockets.AcceptWebSocketAsync();
        var jsonRpc = new JsonRpc(new WebSocketMessageHandler(webSocket));
        jsonRpc.AddLocalRpcTarget(new CalculatorService());
        jsonRpc.StartListening();
        await jsonRpc.Completion;
    }
    else await next();
});
```

**Client (Console)**
```csharp
using StreamJsonRpc;
using System.Net.WebSockets;

var ws = new ClientWebSocket();
await ws.ConnectAsync(new Uri("ws://localhost:5000"), CancellationToken.None);
var rpc = JsonRpc.Attach(ws);
var result = await rpc.InvokeAsync<int>("add", new[] { 5, 3 });
Console.WriteLine($"Result: {result}");
```

**Pros:** Human‑readable, works over WebSockets/HTTP, simple.  
**Cons:** Slower than gRPC, no built‑in streaming.

### 7.3 Simple TCP‑Based Custom RPC (Educational)

This example shows the core concepts: marshalling and sending over TCP.

**Server**
```csharp
using System.Net.Sockets;
using System.Text.Json;

TcpListener listener = new TcpListener(System.Net.IPAddress.Any, 8080);
listener.Start();
while (true)
{
    var client = await listener.AcceptTcpClientAsync();
    _ = Task.Run(() => HandleClient(client));
}

async Task HandleClient(TcpClient client)
{
    var stream = client.GetStream();
    var reader = new StreamReader(stream);
    var writer = new StreamWriter(stream) { AutoFlush = true };

    while (true)
    {
        var line = await reader.ReadLineAsync();
        if (line == null) break;
        var request = JsonSerializer.Deserialize<RpcRequest>(line);
        if (request.Method == "add")
        {
            var args = request.Params.ToArray();
            var result = (int)args[0] + (int)args[1];
            var response = new { jsonrpc = "2.0", result, id = request.Id };
            await writer.WriteLineAsync(JsonSerializer.Serialize(response));
        }
    }
}

record RpcRequest(string Method, object[] Params, int Id);
```

**Client**
```csharp
using TcpClient client = new TcpClient("localhost", 8080);
var stream = client.GetStream();
var writer = new StreamWriter(stream) { AutoFlush = true };
var reader = new StreamReader(stream);

var request = new { jsonrpc = "2.0", method = "add", @params = new[] { 5, 3 }, id = 1 };
await writer.WriteLineAsync(JsonSerializer.Serialize(request));
var response = await reader.ReadLineAsync();
Console.WriteLine(response);
```

## 8. Best Practices for RPC

1. **Choose the right RPC type** – gRPC for performance & polyglot; JSON‑RPC for simplicity.
2. **Define contracts clearly** – Use IDL (Protobuf) for gRPC; document JSON‑RPC methods.
3. **Handle errors gracefully** – Return standard error objects (e.g., JSON‑RPC error code).
4. **Implement timeouts and deadlines** – gRPC supports `CancellationToken` and deadlines.
5. **Use authentication** – gRPC supports JWT, mTLS; JSON‑RPC can use API keys or Bearer tokens.
6. **Versioning** – For gRPC, use package versioning in `.proto` or separate services.
7. **Monitor** – Log request/response metadata (not payloads) for debugging.
8. **Avoid large messages** – Use streaming for large data.
9. **Secure transport** – Always use TLS in production.
10. **Test with real clients** – Use gRPCurl or Postman (with gRPC plugin).

---

## 9. Conclusion

RPC remains a powerful pattern for building efficient, low‑latency distributed systems. **gRPC** is the modern standard for microservices and cross‑language communication. **JSON‑RPC** offers simplicity for lightweight scenarios. Understanding the trade‑offs between RPC, REST, and GraphQL helps you choose the right tool for the job.

Use RPC when you need:
- High performance and low overhead.
- Tightly coupled, well‑defined operations.
- Streaming data (gRPC).
- Polyglot environments with strong code generation.

The C# examples provided give you a starting point for implementing both gRPC and JSON‑RPC in your applications.
