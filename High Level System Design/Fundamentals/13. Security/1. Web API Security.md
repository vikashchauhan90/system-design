# Comprehensive Guide to Web API Security Issues

## 1. Improper Authorization <a name="improper-authorization"></a>

### Problem
Improper authorization occurs when an authenticated user is allowed to perform actions or access resources beyond their privileges. Example: a regular user can access an admin endpoint.

### Example (Vulnerable)
```csharp
[ApiController]
[Route("api/admin")]
public class AdminController : ControllerBase
{
    [HttpGet("users")]
    public IActionResult GetAllUsers()
    {
        // No authorization check – any authenticated user can call this
        var users = _db.Users.ToList();
        return Ok(users);
    }
}
```

### Fix
Use explicit authorization attributes (e.g., `[Authorize(Roles = "Admin")]`) or policy‑based checks.

```csharp
[ApiController]
[Route("api/admin")]
[Authorize(Roles = "Admin")]  // Only users with Admin role
public class AdminController : ControllerBase
{
    [HttpGet("users")]
    public IActionResult GetAllUsers()
    {
        var users = _db.Users.ToList();
        return Ok(users);
    }
}
```

For resource‑specific permissions, use custom authorization handlers.

## 2. Object‑Level Access Control <a name="object-level-access-control"></a>

### Problem
A user can access or modify another user’s data by changing an object ID in the request (e.g., `GET /api/orders/123` where 123 belongs to a different user).

### Example (Vulnerable)
```csharp
[HttpGet("{orderId}")]
public IActionResult GetOrder(int orderId)
{
    var order = _db.Orders.Find(orderId);
    // No check that the order belongs to the current user
    return Ok(order);
}
```

### Fix
Add a check that the resource belongs to the authenticated user.

```csharp
[HttpGet("{orderId}")]
public IActionResult GetOrder(int orderId)
{
    var userId = User.FindFirstValue(ClaimTypes.NameIdentifier);
    var order = _db.Orders.FirstOrDefault(o => o.Id == orderId && o.UserId == userId);
    if (order == null) return NotFound();
    return Ok(order);
}
```

For more complex scenarios, use a repository that automatically filters by user context.

## 3. Missing Function‑Level Access Control <a name="missing-function-level-access-control"></a>

### Problem
Some endpoints (e.g., administrative functions) are not protected, and an attacker can guess or discover them (e.g., `/api/admin/deleteUser`).

### Example (Vulnerable)
```csharp
[HttpPost("deleteUser")]
public IActionResult DeleteUser(int userId)
{
    // No authorization attribute
    _db.Users.Remove(_db.Users.Find(userId));
    return Ok();
}
```

### Fix
Apply `[Authorize]` attributes at the controller or action level, and use role‑based or policy‑based requirements.

```csharp
[HttpPost("deleteUser")]
[Authorize(Roles = "Admin")]
public IActionResult DeleteUser(int userId)
{
    _db.Users.Remove(_db.Users.Find(userId));
    return Ok();
}
```

Additionally, avoid exposing internal endpoints unintentionally by using route conventions and API versioning only for intended public endpoints.

## 4. Improper Authentication <a name="improper-authentication"></a>

### Problem
Weak authentication mechanisms (e.g., no token expiry, using Basic Auth without HTTPS, hardcoded credentials) allow attackers to impersonate users.

### Example (Vulnerable)
```csharp
// No authentication at all – anyone can call
[HttpGet("sensitive")]
public IActionResult SensitiveData() { ... }

// Or using a static API key sent in query string
[HttpGet("data")]
public IActionResult GetData(string apiKey) 
{
    if (apiKey == "static-key-123") return Ok("secret");
}
```

### Fix
Implement standard authentication (JWT, OAuth2, Identity). Enforce HTTPS, use short‑lived tokens, and rotate secrets.

**ASP.NET Core JWT example:**
```csharp
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidIssuer = "your-issuer",
            ValidAudience = "your-audience",
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes("a-very-long-secret-key"))
        };
    });
app.UseAuthentication();
app.UseAuthorization();
```

Apply `[Authorize]` to protected endpoints.

## 5. XXE (XML External Entity) Attack <a name="xxe-attack"></a>

### Problem
If your API accepts XML input and uses an insecure XML parser, an attacker can include external entities to read local files, perform SSRF, or cause denial of service.

### Example (Vulnerable)
```csharp
[HttpPost("upload")]
public IActionResult UploadXml([FromBody] string xml)
{
    XmlDocument doc = new XmlDocument();
    doc.LoadXml(xml);  // DTD processing enabled by default – vulnerable
    // ...
}
```

### Fix
Disable DTD processing and external entity resolution.

**For `XmlDocument` (legacy):**
```csharp
XmlDocument doc = new XmlDocument();
doc.XmlResolver = null;  // Disable DTD
doc.LoadXml(xml);
```

**For `XmlReader` (recommended):**
```csharp
var settings = new XmlReaderSettings
{
    DtdProcessing = DtdProcessing.Prohibit,
    XmlResolver = null
};
using var reader = XmlReader.Create(new StringReader(xml), settings);
var doc = XDocument.Load(reader);
```

**Better yet, avoid XML if possible; use JSON.**

## 6. Vulnerable Components <a name="vulnerable-components"></a>

### Problem
Using outdated NuGet packages, JavaScript libraries, or third‑party components with known vulnerabilities (e.g., Log4j, old Newtonsoft.Json with security flaws).

### Example
Your `*.csproj` references an old version of a package with a CVE:
```xml
<PackageReference Include="Newtonsoft.Json" Version="12.0.1" />
```

### Fix
Regularly scan dependencies with tools like `dotnet list package --vulnerable`, OWASP Dependency Check, or GitHub Dependabot. Keep packages updated.

```bash
dotnet list package --vulnerable
dotnet update package Newtonsoft.Json
```

Use a software composition analysis (SCA) tool in CI/CD.

## 7. DOM‑Based Cross‑Site Scripting (XSS) <a name="dom-based-xss"></a>

### Problem
In Web APIs, XSS typically occurs when an API returns user‑supplied data without proper encoding, and that data is later rendered in a browser (e.g., in a SPA). DOM‑based XSS happens when client‑side JavaScript unsafely writes to the DOM.

### Example (API returns unsafe HTML)
```csharp
[HttpGet("comment")]
public IActionResult GetComment(int id)
{
    var comment = _db.Comments.Find(id); // comment.Text = "<script>alert(1)</script>"
    return Ok(new { text = comment.Text });
}
```
If the frontend does: `element.innerHTML = response.text;` → XSS.

### Fix
- Always encode output based on context (HTML, JavaScript, URL). Use a library like `System.Text.Encodings.Web`.
- Set `Content-Security-Policy` headers to restrict script execution.
- Use frameworks that auto‑encode (React, Angular).

**API should return raw data and let frontend handle encoding safely:**
```csharp
return Ok(new { text = comment.Text }); // still raw, but frontend must use .textContent not .innerHTML
```

**Add Content‑Security‑Policy header:**
```csharp
app.Use(async (context, next) =>
{
    context.Response.Headers.Add("Content-Security-Policy", "default-src 'self'");
    await next();
});
```

## 8. Clickjacking <a name="clickjacking"></a>

### Problem
Clickjacking (UI redress attack) tricks a user into clicking an invisible iframe loaded from your API’s frontend. The attacker overlays transparent buttons to perform actions without user consent.

### Fix
Set the `X-Frame-Options` header to prevent your site from being embedded in iframes, or use `Content-Security-Policy: frame-ancestors`.

**In ASP.NET Core:**
```csharp
app.Use(async (context, next) =>
{
    context.Response.Headers.Add("X-Frame-Options", "DENY");
    await next();
});
```

Or via CSP:
```csharp
context.Response.Headers.Add("Content-Security-Policy", "frame-ancestors 'none'");
```

For specific trusted origins:
```csharp
context.Response.Headers.Add("Content-Security-Policy", "frame-ancestors https://trusted.com");
```

## 9. Using Components from Untrusted Sources <a name="untrusted-components"></a>

### Problem
Including third‑party libraries, scripts, or CDNs without verifying integrity exposes your API frontend to supply‑chain attacks. A compromised CDN could inject malicious code.

### Example
```html
<script src="https://cdn.example.com/library.js"></script>
```

### Fix
- Use Subresource Integrity (SRI) to ensure the loaded script matches a known hash.
- Host critical libraries locally.
- Audit all third‑party dependencies.

**SRI example:**
```html
<script src="https://cdn.example.com/library.js"
        integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8wC"
        crossorigin="anonymous"></script>
```

For .NET applications, prefer NuGet packages from trusted sources (nuget.org) and validate package signatures.

## 10. DDoS (Distributed Denial of Service) <a name="ddos"></a>

### Problem
Attackers flood your API with requests, causing resource exhaustion and service unavailability.

### Fix
Implement rate limiting, request size limits, and timeouts. Use a Web Application Firewall (WAF) or DDoS protection service (Cloudflare, Azure DDoS Protection).

**Rate limiting in ASP.NET Core (using `AspNetCoreRateLimit` or built‑in .NET 7+ `RateLimiter`):**

```csharp
builder.Services.AddRateLimiter(options =>
{
    options.GlobalLimiter = PartitionedRateLimiter.Create<HttpContext, string>(
        httpContext => RateLimitPartition.GetFixedWindowLimiter(
            partitionKey: httpContext.User.Identity?.Name ?? httpContext.Connection.RemoteIpAddress?.ToString(),
            factory: partition => new FixedWindowRateLimiterOptions
            {
                AutoReplenishment = true,
                PermitLimit = 100,
                QueueLimit = 0,
                Window = TimeSpan.FromMinutes(1)
            }));
    options.OnRejected = (context, token) =>
    {
        context.HttpContext.Response.StatusCode = 429;
        return ValueTask.CompletedTask;
    };
});
app.UseRateLimiter();
```

**Request size limit:**
```csharp
builder.Services.Configure<FormOptions>(options =>
{
    options.ValueLengthLimit = 2048;
    options.MultipartBodyLengthLimit = 1024 * 1024; // 1 MB
});
```

## 11. CORS (Cross‑Origin Resource Sharing) Misconfiguration <a name="cors-misconfiguration"></a>

### Problem
Overly permissive CORS policies (e.g., `AllowAnyOrigin` combined with `AllowCredentials()`, or using `*` in production) allow malicious websites to read sensitive data from your API.

### Example (Vulnerable)
```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("OpenCors", policy =>
    {
        policy.AllowAnyOrigin()
              .AllowAnyMethod()
              .AllowAnyHeader()
              .AllowCredentials();  // Not allowed with AllowAnyOrigin – will cause runtime error or ignore
    });
});
```

### Fix
Specify exact trusted origins, avoid `AllowAnyOrigin` with credentials, and limit methods/headers.

**Production‑safe configuration:**
```csharp
builder.Services.AddCors(options =>
{
    options.AddPolicy("StrictCors", policy =>
    {
        policy.WithOrigins("https://trustedfrontend.com", "https://admin.trusted.com")
              .WithMethods("GET", "POST")
              .WithHeaders("content-type", "authorization")
              .AllowCredentials();  // only with specific origins
    });
});
app.UseCors("StrictCors");
```

Never use `AllowAnyOrigin` with `AllowCredentials`.

## 12. Cross‑Site Request Forgery (CSRF / Anti‑Forgery) <a name="csrf"></a>

### Problem
An attacker tricks an authenticated user into making an unwanted request to your API (e.g., transferring money) by embedding a hidden form or image in a malicious website. The browser automatically sends cookies (session cookies) with the request.

### Example (Vulnerable)
Your API uses cookie‑based authentication (session cookie). An attacker crafts a form on `evil.com` that POSTs to `https://yourbank.com/api/transfer`.

### Fix
- Use anti‑forgery tokens (for traditional web apps).
- For APIs, avoid relying solely on cookies; use tokens in custom headers (e.g., JWT in `Authorization` header) – these are not automatically sent by browsers.
- If cookies must be used, implement anti‑CSRF tokens.

**ASP.NET Core anti‑forgery (for web apps not SPAs):**
```csharp
services.AddAntiforgery();
[AutoValidateAntiforgeryToken] // globally
// or on action: [ValidateAntiforgeryToken]
```

**For SPA + API:**
- Use JWT in `Authorization` header (not cookie). Browsers won’t auto‑send it cross‑origin.
- If using cookies, set `SameSite=Strict` or `Lax` and require custom header (e.g., `X-CSRF-TOKEN`) validated on the server.

**Example with custom header:**
```csharp
[HttpPost("transfer")]
public IActionResult Transfer(decimal amount)
{
    var csrfHeader = Request.Headers["X-CSRF-TOKEN"].ToString();
    if (!IsValidCsrfToken(csrfHeader)) return Unauthorized();
    // ...
}
```

## 13. Conclusion <a name="conclusion"></a>

Securing a Web API requires a multi‑layered approach. The issues covered in this document represent the most common vulnerabilities (OWASP API Security Top 10). Each problem is accompanied by an example and a concrete fix in C# / ASP.NET Core.

**Summary checklist:**
- ✅ Implement proper authorization (roles/policies).
- ✅ Enforce object‑level access checks.
- ✅ Protect all endpoints with `[Authorize]` where needed.
- ✅ Use strong authentication (JWT, OAuth2).
- ✅ Disable DTD in XML parsers.
- ✅ Keep dependencies up to date.
- ✅ Sanitize output to prevent XSS.
- ✅ Set `X-Frame-Options` or CSP for clickjacking.
- ✅ Validate third‑party components (SRI, trusted sources).
- ✅ Apply rate limiting and request size limits.
- ✅ Configure CORS strictly – never `AllowAnyOrigin` with credentials.
- ✅ Use anti‑CSRF tokens or token‑based auth to prevent CSRF.

By integrating these fixes into your development lifecycle, you can significantly reduce the attack surface of your Web API.
