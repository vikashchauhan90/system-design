# Comprehensive Guide to Application Protocols: REST, RPC, GraphQL, WebSockets


## 1. Introduction

Application protocols define how software components communicate over a network. The four most common paradigms today are **REST**, **RPC**, **GraphQL**, and **WebSockets**. Each has distinct characteristics in terms of data model, transport, coupling, and real‑time capabilities. Understanding them helps architects and developers select the right tool for the job.

---

## 2. REST (Representational State Transfer)

### 2.1 Core Principles

REST is an architectural style (not a protocol) that uses HTTP as its transport. It treats resources (nouns) identified by URLs, and uses HTTP methods (verbs) to manipulate them.

- **Resource‑oriented** – `/users/123`, `/orders`.
- **Stateless** – each request contains all necessary information.
- **Uniform interface** – standard HTTP methods (GET, POST, PUT, PATCH, DELETE).
- **Representations** – JSON, XML, HTML, etc.
- **Hypermedia (HATEOAS)** – links guide client interactions (optional but encouraged).

### 2.2 Strengths and Weaknesses

| Strengths | Weaknesses |
|-----------|------------|
| Simple, well‑understood, widely adopted | Over‑fetching / under‑fetching common |
| Works with any HTTP client (browsers, curl) | No built‑in real‑time push |
| Excellent caching support (HTTP cache) | Multiple round trips for complex data |
| Stateless → scalable | Versioning can be awkward (URLs, headers) |
| Self‑documenting with OpenAPI | Not ideal for fine‑grained operations |

### 2.3 When to Use REST

- Public APIs where simplicity and broad compatibility matter.
- CRUD‑oriented applications.
- Resource‑centric domains (e.g., blog posts, products).
- When you need HTTP caching (CDN, reverse proxy).

### 2.4 C# Example (ASP.NET Core)

```csharp
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private static List<Product> _products = new();

    [HttpGet]
    public IActionResult GetAll() => Ok(_products);

    [HttpGet("{id}")]
    public IActionResult GetById(int id)
    {
        var product = _products.FirstOrDefault(p => p.Id == id);
        return product == null ? NotFound() : Ok(product);
    }

    [HttpPost]
    public IActionResult Create(Product product)
    {
        product.Id = _products.Count + 1;
        _products.Add(product);
        return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
    }

    [HttpPut("{id}")]
    public IActionResult Update(int id, Product updated)
    {
        var existing = _products.FirstOrDefault(p => p.Id == id);
        if (existing == null) return NotFound();
        existing.Name = updated.Name;
        existing.Price = updated.Price;
        return Ok(existing);
    }

    [HttpDelete("{id}")]
    public IActionResult Delete(int id)
    {
        var product = _products.FirstOrDefault(p => p.Id == id);
        if (product == null) return NotFound();
        _products.Remove(product);
        return NoContent();
    }
}
```

## 3. RPC (Remote Procedure Call)

### 3.1 Core Principles

RPC allows a client to call a function on a remote server as if it were local. The focus is on **actions** (verbs), not resources.

- **Function‑oriented** – `createUser()`, `getUser(id)`.
- **Tight coupling** – client and server share the same function signature.
- **Protocols**: gRPC (HTTP/2 + Protobuf), JSON‑RPC, XML‑RPC.
- **Binary or text** – gRPC uses binary Protobuf for efficiency.

### 3.2 Strengths and Weaknesses

| Strengths | Weaknesses |
|-----------|------------|
| Natural mapping to programming languages | Tight coupling – changes break clients |
| Very efficient (gRPC uses HTTP/2 multiplexing) | Less browser‑friendly (gRPC‑web needed) |
| Strongly typed (Protobuf IDL) | No standard for caching |
| Built‑in streaming (gRPC) | Requires code generation |
| Low overhead, high performance | Debugging harder than REST (binary) |

### 3.3 When to Use RPC

- Internal microservices where performance is critical.
- Cross‑language environments (gRPC supports many languages).
- Real‑time streaming (gRPC bi‑directional streaming).
- When you have well‑defined operations (e.g., authentication, payments).

### 3.4 C# Example (gRPC)

**Define .proto file:**
```protobuf
syntax = "proto3";
service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
}
message UserRequest { int32 id = 1; }
message UserResponse { string name = 1; string email = 2; }
```

**Server implementation:**
```csharp
public class UserServiceImpl : UserService.UserServiceBase
{
    public override Task<UserResponse> GetUser(UserRequest request, ServerCallContext context)
    {
        var user = new UserResponse { Name = "Alice", Email = "alice@example.com" };
        return Task.FromResult(user);
    }
}
```

**Register in Program.cs:**
```csharp
app.MapGrpcService<UserServiceImpl>();
```

**Client call:**
```csharp
using var channel = GrpcChannel.ForAddress("https://localhost:5001");
var client = new UserService.UserServiceClient(channel);
var response = await client.GetUserAsync(new UserRequest { Id = 1 });
Console.WriteLine($"Name: {response.Name}");
```

## 4. GraphQL

### 4.1 Core Principles

GraphQL is a query language and runtime that allows clients to request exactly the data they need, nothing more and nothing less.

- **Client‑driven** – client specifies shape of response.
- **Single endpoint** – usually `/graphql`.
- **Strongly typed schema** – introspection, documentation.
- **Operations** – queries (read), mutations (write), subscriptions (real‑time).

### 4.2 Strengths and Weaknesses

| Strengths | Weaknesses |
|-----------|------------|
| Eliminates over‑fetching and under‑fetching | Complexity – schema design, resolvers |
| Single round trip for complex data | Caching harder than REST (no URL per resource) |
| Strong typing and introspection | Performance risks (deep queries, N+1 problem) |
| Rapid frontend iteration | File uploads not standard |
| Real‑time with subscriptions (over WebSockets) | Learning curve |

### 4.3 When to Use GraphQL

- Applications with complex, nested data requirements (e.g., dashboards, mobile apps).
- When frontend teams need flexibility without backend changes.
- Aggregating multiple data sources.
- Real‑time updates (subscriptions).

### 4.4 C# Example (Hot Chocolate)

**Install:** `dotnet add package HotChocolate.AspNetCore`

**Define schema:**
```csharp
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
}

public class Query
{
    public List<Product> GetProducts() => _products;
    public Product? GetProduct(int id) => _products.FirstOrDefault(p => p.Id == id);
}
```

**Register GraphQL in Program.cs:**
```csharp
builder.Services.AddGraphQLServer()
    .AddQueryType<Query>();
app.MapGraphQL();
```

**Query example:**
```graphql
query {
  product(id: 1) {
    name
    price
  }
}
```

**Subscription (real‑time)**
```csharp
public class Subscription
{
    [Subscribe]
    public Product ProductAdded([EventMessage] Product product) => product;
}
```

## 5. WebSockets

### 5.1 Core Principles

WebSocket provides a full‑duplex, persistent connection over a single TCP socket, enabling real‑time, bi‑directional communication.

- **Persistent connection** – handshake via HTTP Upgrade.
- **Full‑duplex** – both sides send messages at any time.
- **Low overhead** – minimal framing (no HTTP headers per message).
- **Text or binary** messages.

### 5.2 Strengths and Weaknesses

| Strengths | Weaknesses |
|-----------|------------|
| True real‑time, low latency | Complex to manage (reconnection, heartbeats) |
| Efficient (no per‑message HTTP overhead) | Not all proxies/firewalls support WebSockets |
| Bi‑directional | No built‑in request‑response pattern (you must implement) |
| Ideal for chat, gaming, live data | Scaling stateful connections is harder |

### 5.3 When to Use WebSockets

- Real‑time applications: chat, live sports, stock tickers.
- Collaborative editing (Google Docs style).
- Online gaming.
- IoT command and control.
- When you need bi‑directional push.

### 5.4 C# Example (SignalR – high‑level abstraction)

SignalR simplifies WebSockets with automatic fallback (long polling, SSE).

**Server Hub:**
```csharp
public class ChatHub : Hub
{
    public async Task SendMessage(string user, string message)
    {
        await Clients.All.SendAsync("ReceiveMessage", user, message);
    }
}
```

**Program.cs:**
```csharp
builder.Services.AddSignalR();
app.MapHub<ChatHub>("/chatHub");
```

**JavaScript client:**
```javascript
const connection = new signalR.HubConnectionBuilder()
    .withUrl("/chatHub")
    .build();
connection.on("ReceiveMessage", (user, msg) => console.log(user, msg));
await connection.start();
await connection.invoke("SendMessage", "Alice", "Hello");
```

**C# client:**
```csharp
var connection = new HubConnectionBuilder()
    .WithUrl("https://localhost:5000/chatHub")
    .Build();
await connection.StartAsync();
await connection.InvokeAsync("SendMessage", "Bob", "Hi");
```

**Raw WebSocket (low‑level):**
```csharp
app.UseWebSockets();
app.Map("/ws", async context =>
{
    var ws = await context.WebSockets.AcceptWebSocketAsync();
    var buffer = new byte[1024];
    while (ws.State == WebSocketState.Open)
    {
        var result = await ws.ReceiveAsync(new ArraySegment<byte>(buffer), CancellationToken.None);
        if (result.MessageType == WebSocketMessageType.Close)
            await ws.CloseAsync(WebSocketCloseStatus.NormalClosure, "", CancellationToken.None);
        else
            await ws.SendAsync(new ArraySegment<byte>(buffer, 0, result.Count), result.MessageType, true, CancellationToken.None);
    }
});
```

## 6. Comparison Table

| Feature | REST | RPC (gRPC) | GraphQL | WebSockets |
|---------|------|------------|---------|-------------|
| **Paradigm** | Resource‑oriented | Action/function‑oriented | Query language | Bidirectional messaging |
| **Transport** | HTTP/1.1, HTTP/2 | HTTP/2 | HTTP/1.1, HTTP/2 | ws://, wss:// |
| **Data format** | JSON, XML, etc. | Protocol Buffers (binary) | JSON | Text or binary |
| **Caching** | Native (HTTP) | Not standard | Difficult (requires custom) | Not applicable |
| **Real‑time** | No (requires polling/webhooks) | Streaming supported | Subscriptions over WebSockets | Yes (native) |
| **Browser support** | Excellent | Via gRPC‑web | Excellent | Excellent |
| **Code generation** | OpenAPI tools | Strong (protoc) | Yes (GraphQL codegen) | Manual or SignalR |
| **Use case** | Public APIs, CRUD | Internal microservices | Complex frontends, mobile | Live chat, gaming, feeds |

## 7. Choosing the Right Protocol <a name="choosing"></a>

| If you need… | Recommended |
|--------------|-------------|
| Simple, cacheable, resource‑oriented API for public consumption | **REST** |
| High‑performance, typed, internal microservices (especially polyglot) | **RPC (gRPC)** |
| Flexible queries, avoid over‑fetching, rapid frontend iteration | **GraphQL** |
| Real‑time, bi‑directional, low latency (chat, gaming) | **WebSockets** (often with SignalR) |
| Real‑time but only server → client (notifications) | **SSE** (not covered in depth here) |
| Mixed: need REST + real‑time | Combine REST + WebSockets |

Many applications use **multiple protocols**: REST for CRUD, GraphQL for complex queries, WebSockets for real‑time, and gRPC for service‑to‑service calls.

---

## 8. Conclusion

Each application protocol addresses different needs:

- **REST** – the default for public, resource‑oriented APIs.
- **RPC (gRPC)** – high‑performance internal communication.
- **GraphQL** – client‑driven, flexible data fetching.
- **WebSockets** – full‑duplex real‑time communication.

Understanding trade‑offs allows you to design systems that are efficient, scalable, and maintainable. In practice, you may use REST for external APIs, gRPC for internal services, GraphQL for a BFF (Backend for Frontend), and WebSockets for live features.

The C# examples provide a starting point for implementing each pattern in .NET. Choose the right tool for the job, and don't hesitate to mix protocols when justified.
