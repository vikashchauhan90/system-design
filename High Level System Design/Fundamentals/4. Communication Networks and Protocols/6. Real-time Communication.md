# Comprehensive Guide to Real-time Communication: WebRTC, Server-Sent Events (SSE), and Long Polling

## 1. Introduction

Real-time communication allows data to be delivered with very low latency, enabling interactive features like chat, live video, gaming, and notifications. Three prominent techniques are:

- **Long Polling** – A fallback technique that simulates push over HTTP.
- **Server-Sent Events (SSE)** – One-way push from server to client over a persistent HTTP connection.
- **WebRTC** – Peer-to-peer audio, video, and data streaming with minimal latency.

This guide explains each technology, when to use it, and provides practical C# (and JavaScript) examples.

---

## 2. Long Polling

Long polling is a technique where the client sends an HTTP request, and the server holds the connection open until new data is available or a timeout occurs. The client then immediately sends a new request.

### 2.1 How It Works

1. Client sends `GET /poll`.
2. Server checks for new data. If none, it waits (e.g., 30 seconds).
3. When data arrives or timeout expires, server responds with the data (or empty).
4. Client processes response and immediately sends another request.

This gives the illusion of a push mechanism over standard HTTP.

### 2.2 Pros and Cons

| Pros | Cons |
|------|------|
| Works with any HTTP client and server | High latency compared to WebSockets |
| No special server configuration | Many open connections consume server resources |
| Firewall friendly | Inefficient (headers repeated per request) |
| Simple to implement | Head‑of‑line blocking per connection |

### 2.3 Use Cases

- Legacy browser support (Internet Explorer).
- Simple real‑time notifications where WebSockets/SSE are not available.
- Low‑frequency updates (e.g., new email indicator).

### 2.4 C# Implementation (ASP.NET Core)

**Server Endpoint**
```csharp
[ApiController]
[Route("api/longpoll")]
public class LongPollController : ControllerBase
{
    private static readonly ConcurrentQueue<string> _messageQueue = new();
    private static readonly List<TaskCompletionSource<string>> _waitingClients = new();

    [HttpPost("send")]
    public IActionResult SendMessage([FromBody] string message)
    {
        _messageQueue.Enqueue(message);
        NotifyWaitingClients();
        return Ok();
    }

    [HttpGet("poll")]
    public async Task<IActionResult> Poll(CancellationToken cancellationToken)
    {
        // Try to dequeue immediately
        if (_messageQueue.TryDequeue(out var immediateMsg))
            return Ok(immediateMsg);

        // Otherwise wait for new message (timeout 30s)
        var tcs = new TaskCompletionSource<string>();
        using var cts = CancellationTokenSource.CreateLinkedTokenSource(cancellationToken);
        cts.CancelAfter(30000);

        _waitingClients.Add(tcs);
        using var registration = cts.Token.Register(() => tcs.TrySetResult(null));

        var result = await tcs.Task;
        _waitingClients.Remove(tcs);

        if (result == null)
            return NoContent(); // timeout

        return Ok(result);
    }

    private static void NotifyWaitingClients()
    {
        while (_messageQueue.TryDequeue(out var msg) && _waitingClients.Count > 0)
        {
            var tcs = _waitingClients[0];
            _waitingClients.RemoveAt(0);
            tcs.TrySetResult(msg);
        }
    }
}
```

**JavaScript Client**
```javascript
async function longPoll() {
    try {
        const response = await fetch('/api/longpoll/poll');
        if (response.status === 200) {
            const data = await response.text();
            console.log('Received:', data);
        }
    } catch (err) {
        console.error(err);
    } finally {
        longPoll(); // immediately start next request
    }
}
longPoll();
```

## 3. Server-Sent Events (SSE)

SSE is a standard (part of HTML5) that allows a server to push data to the client over a single, long‑lived HTTP connection. The client uses the `EventSource` API.

### 3.1 How It Works

1. Client opens a connection to an SSE endpoint (e.g., `/sse`) with `Accept: text/event-stream`.
2. Server responds with `Content-Type: text/event-stream` and keeps the connection open.
3. Server sends messages in a specific format: `data: ...\n\n`.
4. Client listens to `onmessage` or custom events.

### 3.2 Pros and Cons

| Pros | Cons |
|------|------|
| Simple, built‑in browser API (`EventSource`) | One‑way only (server → client) |
| Automatic reconnection | No binary data (must be Base64) |
| Low overhead compared to long polling | Maximum 6 connections per domain (HTTP/1.1) |
| Works over HTTP/1.1 and HTTP/2 | No support in IE (polyfills exist) |

### 3.3 Use Cases

- Live news feeds, stock tickers.
- Server push notifications.
- Real‑time dashboards (e.g., metrics).
- Progress updates for long‑running tasks.

### 3.4 C# Implementation (ASP.NET Core)

**SSE Endpoint**
```csharp
[ApiController]
[Route("api/sse")]
public class SseController : ControllerBase
{
    [HttpGet("stream")]
    public async Task GetStream(CancellationToken cancellationToken)
    {
        Response.Headers.Add("Content-Type", "text/event-stream");
        Response.Headers.Add("Cache-Control", "no-cache");
        Response.Headers.Add("Connection", "keep-alive");

        var i = 0;
        while (!cancellationToken.IsCancellationRequested)
        {
            var data = $"Message {i} at {DateTime.Now:T}";
            await Response.WriteAsync($"data: {data}\n\n");
            await Response.Body.FlushAsync();
            await Task.Delay(2000, cancellationToken);
            i++;
        }
    }
}
```

**JavaScript Client**
```javascript
const source = new EventSource('/api/sse/stream');
source.onmessage = (event) => {
    console.log('SSE:', event.data);
};
source.onerror = (event) => {
    console.error('SSE connection lost, will auto-reconnect');
};
```

**Custom event types:**
Server can send:
```csharp
await Response.WriteAsync($"event: ping\ndata: keepalive\n\n");
```
Client:
```javascript
source.addEventListener('ping', (e) => console.log('Ping:', e.data));
```

## 4. WebRTC (Web Real-Time Communication)

WebRTC enables peer‑to‑peer audio, video, and arbitrary data streaming directly between browsers or native apps without a central server (except for signaling and NAT traversal). It is designed for very low latency, high‑quality real‑time communication.

### 4.1 How It Works (Signaling, STUN, TURN)

WebRTC requires three main components:

1. **Signaling server** – Exchanges session descriptions (SDP) and ICE candidates between peers. This server does not handle media; it just helps peers find each other. (You can implement it with WebSockets, HTTP, etc.)
2. **STUN server** – Discovers the public IP address and port of a peer behind NAT.
3. **TURN server** – Relays media when direct peer‑to‑peer connection fails (e.g., symmetric NATs).

**Flow**:
- Peer A creates an offer (SDP) → sends via signaling server to Peer B.
- Peer B creates an answer → sends back.
- Both exchange ICE candidates (network candidates) via signaling.
- Once candidates match, peers establish a direct P2P connection (or via TURN).

### 4.2 Pros and Cons

| Pros | Cons |
|------|------|
| Ultra‑low latency (direct P2P) | Complex to set up (signaling, STUN/TURN) |
| No media server costs (for 1:1) | Requires TURN for difficult NATs (adds cost/latency) |
| Secure (mandatory DTLS encryption) | Not designed for large‑scale broadcasting (use SFU/MCU) |
| Supports audio, video, and data channels | Browser support varies (but good now) |
| Works without plugins | Signaling server must be reliable |

### 4.3 Use Cases

- Video conferencing (Zoom, Google Meet – though they often use SFU).
- Live screen sharing.
- Peer‑to‑peer file transfer.
- Real‑time gaming (WebRTC DataChannel).
- IoT device streaming (e.g., security camera).

### 4.4 JavaScript Example (with simple C# signaling server using SignalR)

**Signaling Server (C# SignalR Hub)**
```csharp
public class SignalingHub : Hub
{
    public async Task SendOffer(string targetConnectionId, string offer)
    {
        await Clients.Client(targetConnectionId).SendAsync("ReceiveOffer", Context.ConnectionId, offer);
    }

    public async Task SendAnswer(string targetConnectionId, string answer)
    {
        await Clients.Client(targetConnectionId).SendAsync("ReceiveAnswer", answer);
    }

    public async Task SendIceCandidate(string targetConnectionId, string candidate)
    {
        await Clients.Client(targetConnectionId).SendAsync("ReceiveIceCandidate", candidate);
    }
}
```

**JavaScript Client (Browser)**
```javascript
const configuration = { iceServers: [{ urls: 'stun:stun.l.google.com:19302' }] };
let peerConnection;
let localStream;

// Get user media
navigator.mediaDevices.getUserMedia({ video: true, audio: true })
    .then(stream => {
        localStream = stream;
        document.getElementById('localVideo').srcObject = stream;
    });

async function startCall(targetConnectionId) {
    peerConnection = new RTCPeerConnection(configuration);
    localStream.getTracks().forEach(track => peerConnection.addTrack(track, localStream));

    peerConnection.onicecandidate = (event) => {
        if (event.candidate) {
            signalingHub.invoke('SendIceCandidate', targetConnectionId, JSON.stringify(event.candidate));
        }
    };

    peerConnection.ontrack = (event) => {
        document.getElementById('remoteVideo').srcObject = event.streams[0];
    };

    const offer = await peerConnection.createOffer();
    await peerConnection.setLocalDescription(offer);
    signalingHub.invoke('SendOffer', targetConnectionId, JSON.stringify(offer));
}

// SignalR client for signaling
const signalingHub = new signalR.HubConnectionBuilder()
    .withUrl('/signalingHub')
    .build();

signalingHub.on('ReceiveOffer', async (callerId, offer) => {
    if (!peerConnection) {
        peerConnection = new RTCPeerConnection(configuration);
        localStream.getTracks().forEach(track => peerConnection.addTrack(track, localStream));
        peerConnection.onicecandidate = (event) => { /* send to caller */ };
        peerConnection.ontrack = (event) => { /* show remote video */ };
    }
    await peerConnection.setRemoteDescription(JSON.parse(offer));
    const answer = await peerConnection.createAnswer();
    await peerConnection.setLocalDescription(answer);
    await signalingHub.invoke('SendAnswer', callerId, JSON.stringify(answer));
});

signalingHub.on('ReceiveAnswer', async (answer) => {
    await peerConnection.setRemoteDescription(JSON.parse(answer));
});

signalingHub.on('ReceiveIceCandidate', async (candidate) => {
    await peerConnection.addIceCandidate(JSON.parse(candidate));
});

await signalingHub.start();
```

**Note**: For production, you need STUN/TURN servers (e.g., Coturn, Twilio, Xirsys). The example uses Google's public STUN (for testing only).

## 5. Comparison Table

| Feature | Long Polling | SSE | WebRTC |
|---------|--------------|-----|--------|
| **Direction** | Half‑duplex (request/response) | Server → Client only | Full‑duplex (P2P) |
| **Protocol** | HTTP | HTTP (persistent) | UDP (mostly) + TCP |
| **Latency** | High (due to new requests) | Low (push) | Very low (P2P) |
| **Overhead** | High (headers per poll) | Low (single connection) | Very low (no headers) |
| **Browser support** | All | All modern (except IE) | All modern (except IE) |
| **Firewall friendly** | Yes | Yes | Requires STUN/TURN |
| **Binary data** | Base64 | Base64 | Native (ArrayBuffer) |
| **Automatic reconnection** | Manual | Built‑in (EventSource) | Manual (library help) |
| **Server resources** | Many open connections | Fewer than long polling | Low (only signaling) |
| **Media streaming** | No | No | Yes (audio/video) |

## 6. Choosing the Right Technology

| If you need… | Choose |
|--------------|--------|
| One‑way server updates (notifications, feeds) and simple implementation | **SSE** |
| Real‑time bi‑directional communication (chat, live collaboration) with simple messages | **WebSockets** (not covered here) or **WebRTC DataChannel** |
| Low‑latency audio/video calls, screen sharing | **WebRTC** |
| Support for very old browsers (IE) or simple polling with low frequency | **Long Polling** |
| To avoid complex infrastructure but need real‑time push | **SSE** |

**Practical guidance**:
- Use **SSE** for live dashboards, stock tickers, notifications.
- Use **WebRTC** for video calls, peer‑to‑peer file sharing, low‑latency gaming.
- Use **Long Polling** only as a fallback for environments where WebSockets/SSE are unavailable.

In modern applications, **SSE** often replaces long polling for server‑to‑client push, and **WebRTC** or **WebSockets** handle bidirectional needs.

---

## 7. Conclusion

Real‑time communication is essential for modern interactive applications. Long polling is a legacy technique but remains useful in restricted environments. SSE offers a simple, browser‑native one‑way push with automatic reconnection. WebRTC enables high‑performance peer‑to‑peer audio, video, and data streaming with minimal latency.

Each technology has its strengths and weaknesses. By understanding them, you can select the appropriate one for your use case:

- **Long polling** – legacy compatibility.
- **SSE** – simple server‑to‑client real‑time updates.
- **WebRTC** – full‑duplex, low‑latency media and data.

The C# and JavaScript examples provide a foundation to implement these patterns in your applications. For production WebRTC, consider using libraries like `PeerJS` or managed TURN services. For SSE and long polling, ASP.NET Core handles them efficiently.
